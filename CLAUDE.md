## Project overview

Maze generator that exports to PDF, designed to be sent to a reMarkable tablet. The project provides two interfaces — a CLI and a web UI — sharing the same generation core (maze, PDF layout, reMarkable Cloud API client).

**Stack:** Node.js 22 / TypeScript, npm workspaces (monorepo), Vitest, Biome
**Type:** Full-stack (CLI + web API/server + shared core)

## Architecture

npm workspaces monorepo with a core (`core`) shared by the CLI and the web UI, to avoid duplicating maze generation logic, PDF layout, and reMarkable Cloud integration.

```
remarkable-maze-generator/
├── packages/
│   ├── core/         # maze generation, PDF rendering, reMarkable Cloud client (shared)
│   ├── cli/          # CLI (Commander) consuming @remarkable-maze-generator/core
│   └── web/          # web backend (Fastify) + static frontend, consuming @remarkable-maze-generator/core
├── tsconfig.base.json # TypeScript config shared by all packages
└── vitest.config.ts   # root test config (covers all packages)
```

### TDD workflow

For every feature or fix:

1. **Write tests first** — describe expected behavior through tests before writing implementation
2. **Confirm tests fail** — run `npm test` and verify new tests fail (red)
3. **Write implementation** — minimum code to make tests pass
4. **Confirm tests pass** — run `npm test` and verify all tests pass (green)
5. **Refactor if needed** — clean up while keeping tests green
6. **Run lint** — run `npm run lint` and fix any issues
7. **Present result** — summarize what was done, what was tested, and the test output

### Bug fix workflow

1. **Reproduce in a test first** — write a failing test that captures the bug
2. Fix the bug
3. Confirm the test passes
4. Present result with the test name that now covers the bug

### Self-correction loop

If tests or lint fail:
- Do NOT ask the user for help
- Diagnose the failure, fix it, re-run
- Repeat until green (max 3 self-correction attempts)
- Only escalate to the user if the failure reveals an ambiguous requirement

## Testing conventions

- Test location: `packages/*/src/**/*.test.ts` (a test file next to the module it covers)
- Test runner: Vitest (`npm test` at the root runs every package)
- One test file per source module
- Test names must describe behavior: `"returns empty list when input is empty"` not `"works"`
- Always cover: happy path + at least 2 edge cases + error/invalid input
- The `core` package has no dependency on any Node-specific runtime (no DOM, no direct FS outside an abstraction) so it stays testable in isolation and shareable between the CLI and the web app

## Constraints

- All code must be written in English: identifiers, comments, commit messages, documentation (README, docs/, code comments). This applies regardless of the language used in conversation with the user.
- Never leave dead code or unused imports
- Never hardcode secrets — use environment variables (notably credentials/tokens for the reMarkable Cloud API)
- Never skip tests to meet a deadline — fix the code instead
- Style is enforced by tooling, not by convention — always run `npm run lint` before presenting results
- The `core` package remains the single place implementing maze generation, PDF rendering, and reMarkable Cloud integration — the CLI and the web UI stay plain consumers (no duplicated business logic)
