# Module: web
**Role:** Web server (Fastify) serving the static web UI and a JSON/PDF/SVG/reMarkable API; the static page lets a user configure a maze, see an image preview, download the matching PDF, and send it directly to their reMarkable Cloud account, with a guided one-time pairing step if not already paired (see ADR 020).
**Files:** `packages/web/src/server.ts`, `packages/web/src/maze-routes.ts`, `packages/web/src/maze-request.ts`, `packages/web/src/maze-form-validation.ts`, `packages/web/src/remarkable-routes.ts`, `packages/web/src/remarkable-credential-store.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`
**Exports:**
- `buildServer(options？: BuildServerOptions): FastifyInstance` — wires a file-based `CredentialStore` (shared default path with the CLI) into the reMarkable routes, then registers all routes below.
- `GET /api/version`
- `POST /api/mazes/generate` (accepts `width`/`height`/`seed`/`difficulty`/`solution` in the JSON body, returns the generated PDF directly with `Content-Type: application/pdf`, or a 400 JSON `{ error }` on invalid parameters or an invalid `solution` value — see ADR 017)
- `POST /api/mazes/preview` (accepts the same `width`/`height`/`seed`/`difficulty`, returns an SVG image with `Content-Type: image/svg+xml`, or a 400 JSON `{ error }` — shares its maze-building step with `/api/mazes/generate` to avoid duplicating request parsing, see ADR 019)
- `GET /api/remarkable/status` — `{ authenticated: boolean }`, based on whether credentials are already stored
- `POST /api/remarkable/pair` — accepts `{ pairingCode }`, persists the resulting credentials on success (`{ authenticated: true }`), or 400 `{ error }` when the pairing code is missing or rejected by reMarkable Cloud
- `POST /api/mazes/send` — accepts the same maze parameters as `/api/mazes/generate` plus an optional `visibleName`; regenerates the PDF server-side (stateless, no reuse of an earlier `/generate`/`/preview` call, see ADR 020) and uploads it; returns `{ visibleName }` on success, 400 on invalid maze parameters, 409 `{ error: "not_authenticated" }` when no credentials are stored yet, or 502 `{ error }` when authentication or the upload itself fails; default `visibleName` is `` `${maze.type}-${width}x${height}-${seed}` `` (see ADR 020, generalizing the CLI's ADR 014)
- `buildMazeFromRequest(body): Maze` (request parsing used by both `maze-routes.ts` and `remarkable-routes.ts`); re-exports `core`'s `isValidSolutionMode`/`invalidSolutionModeMessage` for those same call sites, now the single implementation shared with the CLI's own `--solution` validation — see ADR 021
- `createFileCredentialStore(filePath): CredentialStore`, `DEFAULT_CREDENTIALS_PATH` (same default path and file format as the CLI's own store — a device paired via either surface is recognized by both, see ADR 020)
- `validateMazeFormInput(input): MazeFormValidationResult` (tested rules for width/height as positive integers and difficulty 1–5, mirrored by plain JS in `public/app.js` since the static page has no build step — see ADR 018)
- Static files served from `public/`: a maze configuration form with an inline SVG preview, a "Download PDF" link, and a "Send to reMarkable" button (both shown only after a successful generation); sending shows a guided pairing form (one-time code input) when the browser isn't paired yet, and automatically retries the send once pairing succeeds — see ADR 019 (PDF preview) and ADR 020 (send/pairing)
