# Module: web
**Role:** Web server (Fastify) serving the static web UI and a JSON/PDF/SVG/reMarkable API; the static page lets a user configure a maze, see an image preview, download the matching PDF, and send it directly to their reMarkable Cloud account, with a guided one-time pairing step if not already paired (see ADR 020).
**Files:** `packages/web/src/server.ts`, `packages/web/src/process-lifecycle.ts`, `packages/web/src/maze-routes.ts`, `packages/web/src/maze-request.ts`, `packages/web/src/maze-form-validation.ts`, `packages/web/src/maze-form-field-visibility.ts`, `packages/web/src/form-preferences.ts`, `packages/web/src/remarkable-routes.ts`, `packages/web/src/remarkable-credential-store.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`, `packages/web/public/style.css`
**Exports:**
- `buildServer(options？: BuildServerOptions): FastifyInstance` — wires a file-based `CredentialStore` (shared default path with the CLI) into the reMarkable routes, registers `@fastify/helmet` (CSP restricted to `'self'`, plus `data:` for images) before the static-file plugin so every response carries standard security headers, then registers all routes below.
- `resolvePort(env？): number` / `resolveHost(env？): string` — read `PORT`/`HOST` from the environment for the real server startup (not under `NODE_ENV=test`); `resolveHost` defaults to `"127.0.0.1"` (loopback-only) rather than binding every interface, since the server has no authentication layer of its own — see /vibe:review 2026-07-10
- `registerProcessLifecycleHandlers(app): void` (wired at process startup, not under `NODE_ENV=test`) — listens for `SIGTERM`/`SIGINT` and logs an explicit "Received {signal}, shutting down gracefully..." message before closing the Fastify instance and exiting, and listens for `uncaughtException`/`unhandledRejection` to log an explicit "Unexpected error, shutting down the server" message before exiting with a non-zero code; replaces the prior behavior of the process dying silently (just a bare "Terminated" in the terminal) on a stop signal; the underlying `createShutdownHandler`/`createFatalErrorHandler` take injectable logger/close/exit dependencies so the logging and exit-code behavior is unit-tested without touching real OS signals
- `GET /api/version`
- `POST /api/mazes/generate` (accepts `width`/`height`/`seed`/`difficulty`/`type`/`algorithm`/`solution`/`pathLength`/`pathLengthCandidateCount`/`tubeBackgroundFill` in the JSON body, returns the generated PDF directly with `Content-Type: application/pdf`, or a 400 JSON `{ error }` on invalid parameters, an invalid `solution`/`type`/`algorithm`/`pathLength`/`pathLengthCandidateCount` value, an unsupported `type`+`algorithm` combination, or `pathLengthCandidateCount` set without `pathLength` — see ADR 017, ADR 022, ADR 033, ADR 046 and ADR 047; `tubeBackgroundFill` is forwarded as-is to `core`'s `renderMazeToPdf`, filling the tube background for `rectangle-crossing`/`circle-crossing` mazes only — see ADR 060)
- `POST /api/mazes/preview` (accepts the same `width`/`height`/`seed`/`difficulty`/`type`/`algorithm`/`pathLength`/`pathLengthCandidateCount`, returns an SVG image with `Content-Type: image/svg+xml`, or a 400 JSON `{ error }` — shares its maze-building step with `/api/mazes/generate` to avoid duplicating request parsing, see ADR 019; also accepts an optional `showSolution` boolean, kept separate from the PDF endpoints' `solution` enum field — when `true`, the SVG traces the solution path with branch-point markers (via `core`'s `renderMazeToSvg({ showSolution })`) and the response carries an `x-solution-branch-point-count` header from `core`'s `findSolutionBranchPoints`; omitted/`false` keeps the prior response unchanged, no header — see ADR 028; also accepts an optional `tubeBackgroundFill` boolean, forwarded to `core`'s `renderMazeToSvg` the same way as the PDF endpoint — see ADR 060; the response also always carries an `x-maze-seed` header with the maze's resolved seed — random when the request omitted one, or the winning candidate's seed when `pathLength` was set (see ADR 046) — so the caller can pin it on later requests instead of each one independently resolving its own random seed, which previously made the PDF download and the reMarkable upload silently diverge from the preview)
- `GET /api/remarkable/status` — `{ authenticated: boolean }`, based on whether credentials are already stored
- `POST /api/remarkable/pair` — accepts `{ pairingCode }`, persists the resulting credentials on success (`{ authenticated: true }`), or 400 `{ error }` when the pairing code is missing or rejected by reMarkable Cloud
- `POST /api/mazes/send` — accepts the same maze parameters as `/api/mazes/generate` (including `tubeBackgroundFill`) plus an optional `visibleName` and an optional `folder`; regenerates the PDF server-side (stateless, no reuse of an earlier `/generate`/`/preview` call, see ADR 020) and uploads it, into the named reMarkable Cloud folder when `folder` is set (forwarded straight to `core`'s `uploadPdf`, reusing its existing folder resolution — see ADR 013 and backlog item 017) or to the account root otherwise; returns `{ visibleName }` on success, 400 on invalid maze parameters, 409 `{ error: "not_authenticated" }` when no credentials are stored yet, or 502 `{ error }` when authentication or the upload itself fails (including a folder that doesn't exist, surfaced via `core`'s own clear error message); default `visibleName` is `` `${maze.type} ${width}x${height}` `` — no seed (see ADR 020, generalizing the CLI's ADR 014; format changed by ADR 027, scoped to this package only, the CLI's own default name format is unaffected)
- `buildMazeFromRequest(body): Maze` (request parsing used by both `maze-routes.ts` and `remarkable-routes.ts`; forwards `body.type`, `body.algorithm`, `body.pathLength` and `body.pathLengthCandidateCount` to `core`'s `generateMaze`, which rejects an unsupported type/algorithm combination, an invalid `pathLength`, or a `pathLengthCandidateCount` that is non-integer/non-positive, above the cap, or set without `pathLength` — see ADR 047; `body.tubeBackgroundFill` is a rendering-only option, read directly by `maze-routes.ts`/`remarkable-routes.ts` and forwarded to `core`'s `renderMazeToPdf`/`renderMazeToSvg` rather than through this function — see ADR 060); re-exports `core`'s `isValidSolutionMode`/`invalidSolutionModeMessage`/`isValidMazeType`/`invalidMazeTypeMessage`/`isValidMazeAlgorithm`/`invalidMazeAlgorithmMessage` for those same call sites, now the single implementation shared with the CLI's own `--solution`/`--type`/`--algorithm` validation — see ADR 021, ADR 022 and ADR 033
- `createFileCredentialStore(filePath): CredentialStore`, `DEFAULT_CREDENTIALS_PATH` (same default path and file format as the CLI's own store — a device paired via either surface is recognized by both, see ADR 020)
- `validateMazeFormInput(input): MazeFormValidationResult` (tested rules for width/height as positive integers, difficulty 1–5, maze type, generation algorithm, solution display mode, path length target, and path length candidate count — type defaults to `rectangle`, algorithm defaults to `growing-tree`, solution defaults to `none`, path length is left unset (no default — a distinct behavior from any actual target) when omitted, and the candidate count is left unset unless both provided and accompanied by a path length target (integer 1–`MAX_PATH_LENGTH_CANDIDATE_COUNT`, else rejected) — validated via `core`'s shared `isValidMazeType`/`isValidMazeAlgorithm`/`isValidSolutionMode`/`isValidPathLengthTarget`/`MAX_PATH_LENGTH_CANDIDATE_COUNT` — mirrored by plain JS in `public/app.js` since the static page has no build step — see ADR 018, ADR 021, ADR 022, ADR 033, ADR 046 and ADR 047; backlog item 019)
- `computeMazeFormFieldVisibility(input: { type, algorithm, pathLength }): MazeFormFieldVisibility` (derives which of the four conditionally-relevant form fields should be shown, from `core`'s own validation rules rather than a hardcoded ad hoc mapping — `showAlgorithm` is `false` only for the two crossing types, `rectangle-crossing` and `circle-crossing` (each restricted to one algorithm, `growing-tree`, per `validateTypeAlgorithmCompatibility`), `showDifficulty` is `true` only when the *effective* algorithm is `growing-tree` (the only algorithm reading the difficulty knob), `showPathLengthCandidates` is `true` only when a `pathLength` target is set (per ADR 047), and `showTubeBackgroundFill` is `true` only for the same two crossing types, which are the only ones with a tube to fill (per ADR 060); `effectiveAlgorithm` is the algorithm actually used — forced to `growing-tree` for either crossing type regardless of what's stored in the `algorithm` field — mirrored by plain JS in `public/app.js`, same duplication pattern as `validateMazeFormInput`, see ADR 053, ADR 055, ADR 060)
- `serializeFormPreferences(preferences): string`, `parseFormPreferences(rawCookieValue): MazeFormPreferences | null` (encode/decode the eleven remembered form fields — width, height, difficulty, maze type, algorithm, solution mode, show-solution, the reMarkable target folder, path length target, path length candidate count, and the tube background fill checkbox — into/from a single cookie value; parsing rejects a missing, malformed, or wrong-shaped cookie by returning `null` rather than throwing or partially merging, including a cookie written before the folder, path-length, candidate-count, or tube-background-fill fields existed — mirrored by plain JS in `public/app.js`, same duplication pattern as `validateMazeFormInput` — see ADR 042, ADR 043, ADR 046, ADR 047 and ADR 060; backlog item 036)
- Static files served from `public/`: a maze configuration form (width, height, difficulty, a maze type dropdown — classic, bridge crossing, circle, or circle crossing, see ADR 022, ADR 037 and ADR 055 — a generation algorithm dropdown — growing tree, Kruskal, Wilson, or Aldous-Broder, see ADR 033 — a "Solution in PDF" dropdown — none/extra page/overlay, see backlog item 019 — a "Path length" dropdown — none (default)/short/medium/long, see ADR 046 — a "Candidate mazes to compare" number field, only meaningful alongside a path length target (1–50, defaults to 10 when left blank, see ADR 047) — a "Show solution on preview" checkbox, unchecked by default, and a "Fill tube background" checkbox, unchecked by default, only meaningful for the `rectangle-crossing`/`circle-crossing` types (see ADR 060)); the difficulty, generation algorithm, candidate-count, and tube-background-fill fields are shown/hidden live (on `change` of maze type/algorithm/path length, and once on page load) per `computeMazeFormFieldVisibility` — see ADR 053, ADR 060, backlog item 046 — hiding a field never discards its underlying value (so it correctly reappears if its dependency becomes met again, including across a cookie-restored page load) but does override the value actually sent for validation/generation, so a stale hidden value (e.g. a non-`growing-tree` algorithm left over from before switching to `rectangle-crossing`) never reaches `/api/mazes/preview`/`generate`/`send`; with an inline SVG preview, a seed line below it (filled from the preview response's `x-maze-seed` header, e.g. "Seed: 12345", so a mismatch between the preview and what gets downloaded/sent is immediately visible), a branch-point-count line below that (filled from the preview response's `x-solution-branch-point-count` header, cleared when the option is off — see ADR 028), a "Download PDF" link, an optional "reMarkable folder" text field, and a "Send to reMarkable" button (all shown only after a successful generation); the form's own values never include a seed field — the seed is always resolved server-side by the preview call (the winning candidate's seed when a path length target was requested, see ADR 046), then `app.js` re-injects that exact seed — with `pathLength` and `pathLengthCandidateCount` explicitly dropped (the latter requires the former, so both must go together, see ADR 047) — into the request bodies used for the subsequent PDF-generate call and for the one stored for "Send to reMarkable", so those calls replay that exact deterministic seed instead of re-running the candidate search from it as a new base, and the preview, the PDF download, and the reMarkable upload always render the exact same maze instead of each independently resolving its own random seed; the same validated form values (including the solution mode) are otherwise reused as-is for all three calls, so the PDF download and the reMarkable upload always reflect the selected solution mode consistently; the folder field's value is merged into the `/api/mazes/send` request body (omitted when left blank, so sending still goes to the account root); sending shows a guided pairing form (one-time code input) when the browser isn't paired yet, and automatically retries the send once pairing succeeds — see ADR 019 (PDF preview) and ADR 020 (send/pairing); on successful submission, and also when clicking "Send to reMarkable" (via a shared `persistFormPreferences()` helper in `app.js`, so the folder is captured even without re-submitting the form), `app.js` writes the eleven remembered fields — including the reMarkable folder, path length target, path length candidate count, and the tube background fill checkbox, never credentials, and never the seed, which stays per-generation — into a 1-year `maze-form-preferences` cookie via `serializeFormPreferences`, and pre-fills the form (including the folder, path length, candidate count, and tube-background-fill fields) from that cookie via `parseFormPreferences` on the next page load, falling back silently to the hardcoded defaults when no cookie is present or it fails to parse — see ADR 042, ADR 043, ADR 046, ADR 047, ADR 060, backlog item 036
- `public/style.css`: dedicated stylesheet giving the page a coherent, responsive design (form, error message, preview, download/send buttons, pairing form) across the full range from mobile to wide desktop screens (see ADR 054, backlog item 034) — a `min-width: 1000px` breakpoint turns `.page` into a two-column grid (form card left, preview/actions card right) instead of a narrow centered column, a `max-width: 900px` intermediate breakpoint collapses the 5-column form-fields grid to 2 columns before the existing `max-width: 600px` mobile breakpoint collapses it further to 1 column and stacks the action buttons, and buttons/inputs/selects get a `min-height: 44px` for comfortable mobile tap targets; purely presentational, no change to `app.js` behavior, guarded by `public-page.test.ts` (stylesheet is linked and non-empty, required element ids are preserved, all three breakpoints and the two-column wide layout are present, accessibility roles on the error/status messages are preserved)
