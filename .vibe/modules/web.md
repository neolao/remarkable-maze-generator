# Module: web
**Role:** Web server (Fastify) serving the static web UI and a JSON/PDF/SVG/reMarkable API; the static page lets a user configure a maze, see an image preview, download the matching PDF, and send it directly to their reMarkable Cloud account, with a guided one-time pairing step if not already paired (see ADR 020).
**Files:** `packages/web/src/server.ts`, `packages/web/src/maze-routes.ts`, `packages/web/src/maze-request.ts`, `packages/web/src/maze-form-validation.ts`, `packages/web/src/form-preferences.ts`, `packages/web/src/remarkable-routes.ts`, `packages/web/src/remarkable-credential-store.ts`, `packages/web/public/index.html`, `packages/web/public/app.js`, `packages/web/public/style.css`
**Exports:**
- `buildServer(options？: BuildServerOptions): FastifyInstance` — wires a file-based `CredentialStore` (shared default path with the CLI) into the reMarkable routes, then registers all routes below.
- `GET /api/version`
- `POST /api/mazes/generate` (accepts `width`/`height`/`seed`/`difficulty`/`type`/`algorithm`/`solution` in the JSON body, returns the generated PDF directly with `Content-Type: application/pdf`, or a 400 JSON `{ error }` on invalid parameters, an invalid `solution`/`type`/`algorithm` value, or an unsupported `type`+`algorithm` combination — see ADR 017, ADR 022 and ADR 033)
- `POST /api/mazes/preview` (accepts the same `width`/`height`/`seed`/`difficulty`/`type`/`algorithm`, returns an SVG image with `Content-Type: image/svg+xml`, or a 400 JSON `{ error }` — shares its maze-building step with `/api/mazes/generate` to avoid duplicating request parsing, see ADR 019; also accepts an optional `showSolution` boolean, kept separate from the PDF endpoints' `solution` enum field — when `true`, the SVG traces the solution path with branch-point markers (via `core`'s `renderMazeToSvg({ showSolution })`) and the response carries an `x-solution-branch-point-count` header from `core`'s `findSolutionBranchPoints`; omitted/`false` keeps the prior response unchanged, no header — see ADR 028)
- `GET /api/remarkable/status` — `{ authenticated: boolean }`, based on whether credentials are already stored
- `POST /api/remarkable/pair` — accepts `{ pairingCode }`, persists the resulting credentials on success (`{ authenticated: true }`), or 400 `{ error }` when the pairing code is missing or rejected by reMarkable Cloud
- `POST /api/mazes/send` — accepts the same maze parameters as `/api/mazes/generate` plus an optional `visibleName` and an optional `folder`; regenerates the PDF server-side (stateless, no reuse of an earlier `/generate`/`/preview` call, see ADR 020) and uploads it, into the named reMarkable Cloud folder when `folder` is set (forwarded straight to `core`'s `uploadPdf`, reusing its existing folder resolution — see ADR 013 and backlog item 017) or to the account root otherwise; returns `{ visibleName }` on success, 400 on invalid maze parameters, 409 `{ error: "not_authenticated" }` when no credentials are stored yet, or 502 `{ error }` when authentication or the upload itself fails (including a folder that doesn't exist, surfaced via `core`'s own clear error message); default `visibleName` is `` `${maze.type} ${width}x${height}` `` — no seed (see ADR 020, generalizing the CLI's ADR 014; format changed by ADR 027, scoped to this package only, the CLI's own default name format is unaffected)
- `buildMazeFromRequest(body): Maze` (request parsing used by both `maze-routes.ts` and `remarkable-routes.ts`; forwards `body.type` and `body.algorithm` to `core`'s `generateMaze`, which rejects an unsupported combination of the two); re-exports `core`'s `isValidSolutionMode`/`invalidSolutionModeMessage`/`isValidMazeType`/`invalidMazeTypeMessage`/`isValidMazeAlgorithm`/`invalidMazeAlgorithmMessage` for those same call sites, now the single implementation shared with the CLI's own `--solution`/`--type`/`--algorithm` validation — see ADR 021, ADR 022 and ADR 033
- `createFileCredentialStore(filePath): CredentialStore`, `DEFAULT_CREDENTIALS_PATH` (same default path and file format as the CLI's own store — a device paired via either surface is recognized by both, see ADR 020)
- `validateMazeFormInput(input): MazeFormValidationResult` (tested rules for width/height as positive integers, difficulty 1–5, maze type, generation algorithm, and solution display mode — type defaults to `rectangle`, algorithm defaults to `growing-tree`, and solution defaults to `none` when omitted, validated via `core`'s shared `isValidMazeType`/`isValidMazeAlgorithm`/`isValidSolutionMode` — mirrored by plain JS in `public/app.js` since the static page has no build step — see ADR 018, ADR 021, ADR 022, and ADR 033; backlog item 019)
- `serializeFormPreferences(preferences): string`, `parseFormPreferences(rawCookieValue): MazeFormPreferences | null` (encode/decode the seven remembered form fields — width, height, difficulty, maze type, algorithm, solution mode, show-solution — into/from a single cookie value; parsing rejects a missing, malformed, or wrong-shaped cookie by returning `null` rather than throwing or partially merging — mirrored by plain JS in `public/app.js`, same duplication pattern as `validateMazeFormInput` — see ADR 042; backlog item 036)
- Static files served from `public/`: a maze configuration form (width, height, difficulty, a maze type dropdown — classic, bridge crossing, or circle, see ADR 022 and ADR 037 — a generation algorithm dropdown — growing tree, Kruskal, Wilson, or Aldous-Broder, see ADR 033 — a "Solution in PDF" dropdown — none/extra page/overlay, see backlog item 019 — and a "Show solution on preview" checkbox, unchecked by default) with an inline SVG preview, a branch-point-count line below it (filled from the preview response's `x-solution-branch-point-count` header, cleared when the option is off — see ADR 028), a "Download PDF" link, an optional "reMarkable folder" text field, and a "Send to reMarkable" button (all shown only after a successful generation); the same validated form values (including the solution mode) are reused as the request body for the preview, generate, and send calls, so the PDF download and the reMarkable upload always reflect the selected solution mode consistently; the folder field's value is merged into the `/api/mazes/send` request body (omitted when left blank, so sending still goes to the account root); sending shows a guided pairing form (one-time code input) when the browser isn't paired yet, and automatically retries the send once pairing succeeds — see ADR 019 (PDF preview) and ADR 020 (send/pairing); on successful submission, `app.js` also writes the seven remembered fields (not the folder, never credentials) into a 1-year `maze-form-preferences` cookie via `serializeFormPreferences`, and pre-fills the form from that cookie via `parseFormPreferences` on the next page load, falling back silently to the hardcoded defaults when no cookie is present or it fails to parse — see ADR 042, backlog item 036
- `public/style.css`: dedicated stylesheet giving the page a coherent, responsive design (form, error message, preview, download/send buttons, pairing form), with a mobile breakpoint that stacks form fields into a single column; purely presentational, no change to `app.js` behavior, guarded by `public-page.test.ts` (stylesheet is linked and non-empty, required element ids are preserved, a mobile media query exists, accessibility roles on the error/status messages are preserved)
