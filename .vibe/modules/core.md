# Module: core
**Role:** Shared core carrying maze generation, PDF/image rendering, and the reMarkable Cloud client; consumed by `cli` and `web`. Maze generation (with a tunable 1‚Äì5 difficulty level controlling branch-point density via a growing-tree selection bias, see ADR 015; and a choice of maze type ‚Äî classic `rectangle` (thin walls) or `rectangle-crossing`, rendered as a real hollow tube (two independent edge lines per corridor, computed as one polygon boundary per cell ‚Äî see ADR 026, superseding ADR 023's border-fill technique and ADR 025's single-centerline technique), some of which tunnel through another's straight passage forming a real, walkable bridge crossing, not a decorative one ‚Äî see ADR 022, ADR 024 (supersedes ADR 022's decorative-only design)), solution path computation (axis-aware at a crossing cell ‚Äî never turns from one passage onto the other, see ADR 024), PDF rendering (reMarkable 2 page size, with a visible opening at the entrance and exit ‚Äî see ADR 010), SVG image rendering for lightweight previews (see ADR 019), and reMarkable Cloud authentication + upload are implemented on top of the `rmapi-js` library (verified against a real reMarkable account ‚Äî see ADR 012, which supersedes the earlier hand-rolled protocol from ADR 007/008).
**Files:** `packages/core/src/index.ts`, `packages/core/src/maze.ts`, `packages/core/src/maze-solver.ts`, `packages/core/src/maze-layout.ts`, `packages/core/src/maze-pdf.ts`, `packages/core/src/maze-svg.ts`, `packages/core/src/remarkable-auth.ts`, `packages/core/src/remarkable-credential-store.ts`, `packages/core/src/remarkable-upload.ts`
**Exports:** `CORE_VERSION: string`, `generateMaze(options: GenerateMazeOptions): Maze` (`difficulty` is optional, integer 1‚Äì5, defaults to 1 ‚Äî see ADR 015; `type` is optional, defaults to `"rectangle"` ‚Äî see ADR 022; for `rectangle-crossing`, the growing-tree loop's candidate moves include tunneling through an already-visited cell's straight passage to reach new territory, in addition to the normal move to an adjacent unvisited cell ‚Äî both stay real spanning-tree edges, preserving the single-solution guarantee, see ADR 024; the returned `Maze` carries its own resolved `type`, `seed`, `difficulty`, and `crossings` ‚Äî see ADR 016 and ADR 024), `generateMazeBatch(options: GenerateMazeBatchOptions): Maze[]` (derives `seed + index` per maze, forwards `difficulty` and `type` to every maze, each returned maze keeps its own resolved seed), `MAZE_TYPES: MazeType[]`, `isValidMazeType(value: string): value is MazeType`, `invalidMazeTypeMessage(value: string): string` (maze-type validation shared by `cli` and `web`, same pattern as `SOLUTION_MODES` ‚Äî see ADR 022), `solveMaze(maze: Maze): MazePosition[]` (walks `Cell.walls`; at a crossing cell, a path may continue along the axis it entered on but never turn onto the other ‚Äî modeled internally by splitting a crossing cell into two graph nodes, one per axis, without changing the public `MazePosition` shape, see ADR 024), `findSolutionBranchPoints(maze: Maze): MazePosition[]` (path cells with more than the two directions used to arrive and leave ‚Äî i.e. a real alternative existed; entrance/exit excluded; a bridge-crossing cell is never flagged even with all four walls open, since the solver's own axis lock already limits it to the entered axis ‚Äî see ADR 024 and ADR 028), `computeWallSegments(maze: Maze): LineSegment[]` (internal-facing geometry for the classic `rectangle` type ‚Äî unit cell coordinates, encodes the entrance/exit opening rules once so PDF and SVG rendering can't visually drift apart), `computeTubeSegments(maze: Maze): TubeSegment[]` (internal-facing; both edge lines of every corridor in a `rectangle-crossing` maze, computed per cell as the boundary of a small central "hub" plus an "arm" per open side ‚Äî a closed side gets a flat cap, a straight-through pair of opposite open sides gets one uninterrupted full-cell-length pair of lines, and a crossing's over-axis is likewise uninterrupted while its under-axis arms stop at the hub without crossing it, leaving a real gap; adjacent cells' arms always meet at the exact same shared-boundary point, so turns and straight runs connect with zero gap using only independent line segments, see ADR 026; the hub/arm half-width is `TUBE_HALF_WIDTH_RATIO` ‚Äî `0.35` of the cell size, wide enough that the tube takes up more room than the walls/gaps around it, see ADR 029; `TubeSegment` is `LineSegment | ArcSegment` ‚Äî every "real" hub corner (the two adjacent sides it touches are in the *same* open/closed state ‚Äî both open, as at a turn or a full 4-way junction, or both closed, as at a dead end's cap) is replaced by a real `ArcSegment` of radius `TUBE_CORNER_RADIUS_RATIO`, shortening the adjoining lines by that radius; a corner whose two adjacent sides differ (one open, one closed) is already collinear with its neighbor and stays untouched, and a cell with no open side at all (never produced by a real generated maze) is left fully sharp; crossings keep their exact gap, never rounded; `isArcSegment()` is the exported type guard ‚Äî see ADR 030 (turns only) generalized by ADR 031 (every corner shape)), `renderMazeToPdf(maze: Maze, options?: RenderMazeToPdfOptions): Promise<Uint8Array>` (solution can be shown via `"extra-page"` or `"overlay"`, default `"none"`; prints a small footer with the maze's type/dimensions/seed/difficulty when `maze.seed` is set ‚Äî see ADR 016; for `rectangle-crossing`, draws the two tube edge lines with round caps instead of thin walls, see ADR 026), `renderMazeBatchToPdf(mazes: Maze[], options?): Promise<Uint8Array>` (one combined PDF, one maze per page), `renderMazeBatchToPdfs(mazes: Maze[], options?): Promise<Uint8Array[]>` (one separate PDF per maze), `renderMazeToSvg(maze: Maze, options?: RenderMazeToSvgOptions): string` (self-contained SVG markup; thin walls for `rectangle`, hollow-tube `stroke-linecap="round"` edge lines for `rectangle-crossing` ‚Äî used for the web preview so it renders in any browser without depending on a PDF viewer, see ADR 019, ADR 026; `options.showSolution` optionally traces the solution path in red plus a red circle on every branch point found by `findSolutionBranchPoints`, kept as a separate boolean from the PDF's `SolutionDisplayMode` ‚Äî see ADR 028), `SOLUTION_MODES: SolutionDisplayMode[]`, `isValidSolutionMode(value: string): value is SolutionDisplayMode`, `invalidSolutionModeMessage(value: string): string` (solution-mode validation shared by `cli` and `web` so both interfaces enforce and describe the same allowed values ‚Äî see ADR 021), `authenticate(store, pairingCode, options?): Promise<RemarkableSession>` (`RemarkableSession` is now an opaque `rmapi-js` client instance; `CredentialStore` is an interface only, no built-in implementation ‚Äî see ADR 007), `uploadPdf(session, filePath, visibleName, options): Promise<void>` (`options.readFile` is required, no default ‚Äî keeps `core` free of Node-specific imports, see ADR 008; `options.folder` optionally targets an existing reMarkable Cloud folder by name ‚Äî throws a clear error if it doesn't exist, no auto-create, see ADR 013), `REMARKABLE_2_PAGE_WIDTH_PT`, `REMARKABLE_2_PAGE_HEIGHT_PT`, types `Maze`, `Cell`, `CellWalls`, `MazeType`, `MazeCrossing`, `GenerateMazeOptions`, `GenerateMazeBatchOptions`, `MazePosition`, `RenderMazeToPdfOptions`, `SolutionDisplayMode`, `RenderMazeToSvgOptions`, `CredentialStore`, `RemarkableCredentials`, `RemarkableAuthOptions`, `RemarkableSession`, `UploadPdfOptions`
**Depends on:** `pdf-lib` (external, PDF generation), `rmapi-js` (external, reMarkable Cloud protocol ‚Äî requires Node.js 25+, see ADR 012)
