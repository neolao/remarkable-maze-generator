# Module: core
**Role:** Shared core carrying maze generation, PDF/image rendering, and the reMarkable Cloud client; consumed by `cli` and `web`. Maze generation is pluggable across 4 selectable algorithms (`MazeAlgorithm`, default `"growing-tree"`) each isolated in its own module under `maze-algorithms/` so one can be re-tuned without risking the others' behavior — `growing-tree` (the original algorithm, difficulty-tunable, the only one supporting bridge crossings), `kruskal` (randomized Kruskal's via union-find over shuffled edges), `wilson` (loop-erased random walk, a true uniform spanning tree), and `aldous-broder` (plain random walk until every cell is visited, also a true uniform spanning tree but slower) — see ADR 033 (the difficulty knob and bridge crossings are not yet supported by the 3 newer algorithms, a deliberate scoping choice pending real usage feedback). Maze generation (with a tunable 1–5 difficulty level controlling branch-point density via a growing-tree selection bias, see ADR 015; a minimum branch-commit length keeps dead-end branches meaningfully long at every difficulty instead of getting boxed in short after a random jump, independent of the difficulty-to-branch-point-density relationship, see ADR 032; and a choice of maze type — classic `rectangle` (thin walls), `rectangle-crossing`, rendered as a real hollow tube (two independent edge lines per corridor, computed as one polygon boundary per cell — see ADR 026, superseding ADR 023's border-fill technique and ADR 025's single-centerline technique), some of which tunnel through another's straight passage forming a real, walkable bridge crossing, not a decorative one — see ADR 022, ADR 024 (supersedes ADR 022's decorative-only design), or `circle` — a real growing-sector ("theta") topology, entirely separate from the rectangular grid: not a uniform 2D array, since the number of sectors grows ring by ring (starting at `width` sectors at the center, doubling — or more generally multiplying by whatever integer keeps a cell's arc length close to the radial thickness — as needed further out, always an exact multiple of the ring inside it so cell boundaries line up cleanly from one ring to the next) so passage width neither pinches near the center nor balloons near the edge; lives in its own module tree, `circle-maze/` (`topology.ts` for the pure ring/sector math — sector counts, inward-parent/outward-children proportional index mapping, cw/ccw wraparound within a ring; `cells.ts` for the cell/edge data model; `growing-tree.ts`/`kruskal.ts`/`wilson.ts`/`aldous-broder.ts`, each algorithm reimplemented against this graph rather than shared with the rectangular versions; `solve.ts` and `render.ts` for its own solver and polar geometry) — see ADR 037 and ADR 040 (which restored the exact-multiple growth after two intermediate attempts, ADR 038's independent per-ring formula and ADR 039's portal restriction, together failed to keep ring-to-ring cell boundaries aligned — cross-checked against the reference implementation at github.com/codebox/maze.js); the entrance is marked by a "hub" boundary around the center, sized directly from the maze's own starting `width` so its own passage width matches every other ring instead of a fixed, potentially-cramped radius — with a gap at its own sector 0, mirroring the outer boundary's gap at the exit, and a single angle offset rotates both openings to point toward the top of the circle instead of the right; the solution trace follows the radius through a ring transition instead of cutting across at an angle (`computeCircleSolutionPoints`, see ADR 041), solution path computation (axis-aware at a crossing cell — never turns from one passage onto the other, see ADR 024), PDF rendering (reMarkable 2 page size, with a visible opening at the entrance and exit — see ADR 010), SVG image rendering for lightweight previews (see ADR 019), and reMarkable Cloud authentication + upload are implemented on top of the `rmapi-js` library (verified against a real reMarkable account — see ADR 012, which supersedes the earlier hand-rolled protocol from ADR 007/008).
