# Module: core
**Role:** Shared core carrying maze generation, PDF/image rendering, and the reMarkable Cloud client; consumed by `cli` and `web`. Maze generation (with a tunable 1–5 difficulty level controlling branch-point density via a growing-tree selection bias, see ADR 015; and a choice of maze type — classic `rectangle` (thin walls) or `rectangle-crossing`, rendered as thick rounded-cap independent corridor strokes, some of which tunnel through another's straight passage forming a real, walkable bridge crossing, not a decorative one — see ADR 022, ADR 024 (supersedes ADR 022's decorative-only design), and ADR 025 (simple independent strokes, superseding ADR 023's hollow-tube border technique)), solution path computation (axis-aware at a crossing cell — never turns from one passage onto the other, see ADR 024), PDF rendering (reMarkable 2 page size, with a visible opening at the entrance and exit — see ADR 010), SVG image rendering for lightweight previews (see ADR 019), and reMarkable Cloud authentication + upload are implemented on top of the `rmapi-js` library (verified against a real reMarkable account — see ADR 012, which supersedes the earlier hand-rolled protocol from ADR 007/008).
**Files:** `packages/core/src/index.ts`, `packages/core/src/maze.ts`, `packages/core/src/maze-solver.ts`, `packages/core/src/maze-layout.ts`, `packages/core/src/maze-pdf.ts`, `packages/core/src/maze-svg.ts`, `packages/core/src/remarkable-auth.ts`, `packages/core/src/remarkable-credential-store.ts`, `packages/core/src/remarkable-upload.ts`
**Exports:** `CORE_VERSION: string`, `generateMaze(options: GenerateMazeOptions): Maze` (`difficulty` is optional, integer 1–5, defaults to 1 — see ADR 015; `type` is optional, defaults to `"rectangle"` — see ADR 022; for `rectangle-crossing`, the growing-tree loop's candidate moves include tunneling through an already-visited cell's straight passage to reach new territory, in addition to the normal move to an adjacent unvisited cell — both stay real spanning-tree edges, preserving the single-solution guarantee, see ADR 024; the returned `Maze` carries its own resolved `type`, `seed`, `difficulty`, and `crossings` — see ADR 016 and ADR 024), `generateMazeBatch(options: GenerateMazeBatchOptions): Maze[]` (derives `seed + index` per maze, forwards `difficulty` and `type` to every maze, each returned maze keeps its own resolved seed), `MAZE_TYPES: MazeType[]`, `isValidMazeType(value: string): value is MazeType`, `invalidMazeTypeMessage(value: string): string` (maze-type validation shared by `cli` and `web`, same pattern as `SOLUTION_MODES` — see ADR 022), `solveMaze(maze: Maze): MazePosition[]` (walks `Cell.walls`; at a crossing cell, a path may continue along the axis it entered on but never turn onto the other — modeled internally by splitting a crossing cell into two graph nodes, one per axis, without changing the public `MazePosition` shape, see ADR 024), `computeWallSegments(maze: Maze): LineSegment[]` (internal-facing geometry for the classic `rectangle` type — unit cell coordinates, encodes the entrance/exit opening rules once so PDF and SVG rendering can't visually drift apart), `computePathSegments(maze: Maze): LineSegment[]` (internal-facing; corridor-centerline geometry for `rectangle-crossing`'s thick rounded-cap rendering — one segment per open connection between cells, plus an entrance/exit stub to the boundary; excludes only a crossing cell's under-axis connection, see ADR 025), `computeCrossingUnderSegments(maze: Maze): LineSegment[]` (internal-facing; the 2 under-axis segments per entry in `maze.crossings`, each reaching all the way to its real neighbor but stopping just short of the crossing's own center — a real, drawn geometric gap, not a paint-order trick, see ADR 025), `renderMazeToPdf(maze: Maze, options?: RenderMazeToPdfOptions): Promise<Uint8Array>` (solution can be shown via `"extra-page"` or `"overlay"`, default `"none"`; prints a small footer with the maze's type/dimensions/seed/difficulty when `maze.seed` is set — see ADR 016; for `rectangle-crossing`, draws thick rounded-cap (`LineCapStyle.Round`) independent solid strokes instead of thin walls, see ADR 025), `renderMazeBatchToPdf(mazes: Maze[], options?): Promise<Uint8Array>` (one combined PDF, one maze per page), `renderMazeBatchToPdfs(mazes: Maze[], options?): Promise<Uint8Array[]>` (one separate PDF per maze), `renderMazeToSvg(maze: Maze, options?: RenderMazeToSvgOptions): string` (self-contained SVG markup; thin walls for `rectangle`, thick independent `stroke-linecap="round"` corridor strokes for `rectangle-crossing`, no solution overlay — used for the web preview so it renders in any browser without depending on a PDF viewer, see ADR 019, ADR 025), `SOLUTION_MODES: SolutionDisplayMode[]`, `isValidSolutionMode(value: string): value is SolutionDisplayMode`, `invalidSolutionModeMessage(value: string): string` (solution-mode validation shared by `cli` and `web` so both interfaces enforce and describe the same allowed values — see ADR 021), `authenticate(store, pairingCode, options?): Promise<RemarkableSession>` (`RemarkableSession` is now an opaque `rmapi-js` client instance; `CredentialStore` is an interface only, no built-in implementation — see ADR 007), `uploadPdf(session, filePath, visibleName, options): Promise<void>` (`options.readFile` is required, no default — keeps `core` free of Node-specific imports, see ADR 008; `options.folder` optionally targets an existing reMarkable Cloud folder by name — throws a clear error if it doesn't exist, no auto-create, see ADR 013), `REMARKABLE_2_PAGE_WIDTH_PT`, `REMARKABLE_2_PAGE_HEIGHT_PT`, types `Maze`, `Cell`, `CellWalls`, `MazeType`, `MazeCrossing`, `GenerateMazeOptions`, `GenerateMazeBatchOptions`, `MazePosition`, `RenderMazeToPdfOptions`, `SolutionDisplayMode`, `RenderMazeToSvgOptions`, `CredentialStore`, `RemarkableCredentials`, `RemarkableAuthOptions`, `RemarkableSession`, `UploadPdfOptions`
**Depends on:** `pdf-lib` (external, PDF generation), `rmapi-js` (external, reMarkable Cloud protocol — requires Node.js 25+, see ADR 012)
