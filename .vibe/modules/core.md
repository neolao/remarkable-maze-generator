# Module: core
**Role:** Shared core carrying maze generation, PDF/image rendering, and the reMarkable Cloud client; consumed by `cli` and `web`. Maze generation is pluggable across 4 selectable algorithms (`MazeAlgorithm`, default `"growing-tree"`) each isolated in its own module under `maze-algorithms/` so one can be re-tuned without risking the others' behavior — `growing-tree` (the original algorithm, difficulty-tunable, the only one supporting bridge crossings), `kruskal` (randomized Kruskal's via union-find over shuffled edges), `wilson` (loop-erased random walk, a true uniform spanning tree), and `aldous-broder` (plain random walk until every cell is visited, also a true uniform spanning tree but slower) — see ADR 033 (the difficulty knob and bridge crossings are not yet supported by the 3 newer algorithms, a deliberate scoping choice pending real usage feedback). Maze generation (with a tunable 1–5 difficulty level controlling branch-point density via a growing-tree selection bias, see ADR 015; a minimum branch-commit length keeps dead-end branches meaningfully long at every difficulty instead of getting boxed in short after a random jump, independent of the difficulty-to-branch-point-density relationship, see ADR 032; and a choice of maze type — classic `rectangle` (thin walls), `rectangle-crossing`, rendered as a real hollow tube (two independent edge lines per corridor, computed as one polygon boundary per cell — see ADR 026, superseding ADR 023's border-fill technique and ADR 025's single-centerline technique), some of which tunnel through another's straight passage forming a real, walkable bridge crossing, not a decorative one — see ADR 022, ADR 024 (supersedes ADR 022's decorative-only design), or `circle` — the same rectangular grid and all 4 algorithms reused unchanged, laid out in polar coordinates instead (a column becomes an angular sector, a row a concentric ring), with the grid wrapping horizontally end-to-end so the maze reads as a real seamless circle instead of a pie with a permanently walled slice — see ADR 034 (the solver and branch-point detection are wraparound-aware too, since a `circle` maze's boundary walls are real, walkable passages to the opposite column, not the always-closed boundary they are for every other type)), solution path computation (axis-aware at a crossing cell — never turns from one passage onto the other, see ADR 024), PDF rendering (reMarkable 2 page size, with a visible opening at the entrance and exit — see ADR 010), SVG image rendering for lightweight previews (see ADR 019), and reMarkable Cloud authentication + upload are implemented on top of the `rmapi-js` library (verified against a real reMarkable account — see ADR 012, which supersedes the earlier hand-rolled protocol from ADR 007/008).

both edge lines of every corridor in a `rectangle-crossing` maze, computed per cell as the boundary of a small central "hub" plus an "arm" per open side — a closed side gets a flat cap, a straight-through pair of opposite open sides gets one uninterrupted full-cell-length pair of lines, and a crossing's over-axis is likewise uninterrupted while its under-axis arms stop at the hub without crossing it, leaving a real gap; `TubeSegment` is `LineSegment | ArcSegment` — every "real" hub corner (the two adjacent sides it touches are in the *same* open/closed state — both open, as at a turn or a full 4-way junction, or both closed, as at a dead end's cap) is replaced by a real `ArcSegment` of radius `TUBE_CORNER_RADIUS_RATIO`, shortening the adjoining lines by that radius; a corner whose two adjacent sides differ (one open, one closed) is already collinear with its neighbor and stays untouched, and a cell with no open side at all (never produced by a real generated maze) is left fully sharp;
