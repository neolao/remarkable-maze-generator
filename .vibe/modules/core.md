# Module: core
**Role:** Shared core carrying maze generation, PDF/image rendering, and the reMarkable Cloud client; consumed by `cli` and `web`. Domain types (`Maze`, `Cell`, `CellWalls`, `MazeCrossing`, `CircleMazeCrossing`, `MazeType`, `MazeAlgorithm`, `PathLengthTarget`, `GenerateMazeOptions`) and their invariants (dimension bounds, difficulty range, type/algorithm compatibility, path-length target/candidate-count rules) are isolated in `maze-domain.ts`, a dedicated domain module with no runtime dependency on generation, rendering, or reMarkable Cloud code (one narrow type-only exception: `CircleCell`, imported to describe `Maze.circleCells`'s own shape ŌĆö see ADR 050); every consumer that only needs the vocabulary (`maze-algorithm-registry.ts`, `maze-algorithms/*`, `maze-solver.ts`, `rendering/maze-layout.ts`, `rendering/maze-pdf.ts`, `rendering/maze-svg.ts`, `rendering/maze-render-strategy.ts`) imports it directly instead of through `maze.ts`, which now holds only `generateMaze`'s generation-orchestration logic (candidate generation, path-length candidate selection) ŌĆö this also removed a prior type-only import cycle between `maze.ts` and `maze-algorithm-registry.ts`. PDF rendering, SVG rendering, and their shared drawing geometry live under their own `rendering/` module boundary ŌĆö depending on the domain module and `maze-solver.ts` but never imported back by them ŌĆö isolating the output/presentation concern from the domain the way DDD calls for (see ADR 051). The reMarkable Cloud client (`authenticate`, `uploadPdf`, `CredentialStore`) similarly lives under its own `infrastructure/remarkable/` module boundary, marking it as an external-system adapter never imported back by the domain module ŌĆö a pure file relocation with no import changes among the three files themselves, since none of them depended on the domain module (see ADR 052). How each maze type is rendered/solved and how each algorithm is generated is centralized in three `Record`-keyed strategy/registry maps ŌĆö `maze-algorithm-registry.ts` (per-`MazeAlgorithm`, both the rectangular and circle generator), `rendering/maze-render-strategy.ts` (per-`MazeType`, shared by `rendering/maze-pdf.ts`/`rendering/maze-svg.ts`, and itself part of the `rendering/` boundary since it exists solely to serve them), and a solver strategy map local to `maze-solver.ts` ŌĆö rather than duplicated conditional checks, so TypeScript's exhaustiveness checking on the `Record` forces every new type/algorithm to be registered in exactly one place per concern (see ADR 049). Maze generation is pluggable across 4 selectable algorithms (`MazeAlgorithm`, default `"growing-tree"`) each isolated in its own module under `maze-algorithms/` so one can be re-tuned without risking the others' behavior ŌĆö `growing-tree` (the original algorithm, difficulty-tunable, the only one supporting bridge crossings), `kruskal` (randomized Kruskal's via union-find over shuffled edges), `wilson` (loop-erased random walk, a true uniform spanning tree), and `aldous-broder` (plain random walk until every cell is visited, also a true uniform spanning tree but slower) ŌĆö see ADR 033 (the difficulty knob and bridge crossings are not yet supported by the 3 newer algorithms, a deliberate scoping choice pending real usage feedback). Maze generation (with a tunable 1ŌĆō5 difficulty level controlling branch-point density via a growing-tree selection bias, see ADR 015; a minimum branch-commit length keeps dead-end branches meaningfully long at every difficulty instead of getting boxed in short after a random jump, independent of the difficulty-to-branch-point-density relationship, see ADR 032; and a choice of maze type ŌĆö classic `rectangle` (thin walls), `rectangle-crossing`, rendered as a real hollow tube (two independent edge lines per corridor, computed as one polygon boundary per cell ŌĆö see ADR 026, superseding ADR 023's border-fill technique and ADR 025's single-centerline technique), some of which tunnel through another's straight passage forming a real, walkable bridge crossing, not a decorative one ŌĆö see ADR 022, ADR 024 (supersedes ADR 022's decorative-only design), `circle` ŌĆö a real growing-sector ("theta") topology, entirely separate from the rectangular grid: not a uniform 2D array, since the number of sectors grows ring by ring (starting at `width` sectors at the center, doubling ŌĆö or more generally multiplying by whatever integer keeps a cell's arc length close to the radial thickness ŌĆö as needed further out, always an exact multiple of the ring inside it so cell boundaries line up cleanly from one ring to the next) so passage width neither pinches near the center nor balloons near the edge; lives in its own module tree, `circle-maze/` (`topology.ts` for the pure ring/sector math ŌĆö sector counts, inward-parent/outward-children proportional index mapping, cw/ccw wraparound within a ring; `cells.ts` for the cell/edge data model; `growing-tree.ts`/`kruskal.ts`/`wilson.ts`/`aldous-broder.ts`, each algorithm reimplemented against this graph rather than shared with the rectangular versions; `solve.ts` and `render.ts` for its own solver and polar geometry) ŌĆö see ADR 037 and ADR 040 (which restored the exact-multiple growth after two intermediate attempts, ADR 038's independent per-ring formula and ADR 039's portal restriction, together failed to keep ring-to-ring cell boundaries aligned ŌĆö cross-checked against the reference implementation at github.com/codebox/maze.js), or `circle-crossing` ŌĆö combines the circle topology with tube-style bridge crossings, the polar equivalent of `rectangle-crossing`: `circle-maze/growing-tree.ts` carves a crossing as a 2-hop tunnel (`current` ŌåÆ `tunnel` ŌåÆ `target`) along one topological direction (cw, ccw, outward, or inward) whenever the middle hop already carries a clean straight-through passage on the *other* axis (tangential: cw+ccw; radial: inward + a single open outward child), reusing `cells.ts`'s `carveEdge` for both hops so the tunneled-through node ends up with both axes open, mirroring the rectangular tunneled-cell shape; recorded as `CircleMazeCrossing { ring, sector, underAxis: "radial" | "tangential" }` on `Maze.circleCrossings`; `circle-maze/solve.ts`'s BFS carries the same axis lock as the rectangular solver (crossing nodes split into one graph node per axis); `circle-maze/render.ts`'s `computeCircleTubeSegments` draws the outline of the maze's whole passage area, described as a filled region of axis-aligned polar rectangles (one hub band per cell ŌĆö flush with the cell's boundary on open tangential sides so neighbors merge, set back by a cap inset on closed ones so two closed neighbors read as two separate rounded pipe caps with a visible gap, the same closed-pipe look as `rectangle-crossing` ŌĆö plus one connector per open radial edge sized from the child's own physical radius, and entrance/exit stubs); the boundary-extraction engine lives in its own pure module, `circle-maze/region-boundary.ts` (`computePolarRegionBoundary(rects, openings, forcedEdges)`), and since the outline of a filled region is closed by construction, floating fragments and dangling tube stubs ŌĆö the recurring bug class of the three prior per-cell-assembly designs (ADR 055/056/057/058, all superseded) ŌĆö are geometrically impossible; both tube types (`rectangle-crossing` and `circle-crossing`) can optionally have their passage area drawn as a light gray fill under the outline (`options.tubeBackgroundFill` on `renderMazeToPdf`/`renderMazeToSvg`, off by default) ŌĆö `rendering/maze-layout.ts`'s `computeTubeFillRects`/`fillRectToClosedShape` and `circle-maze/render.ts`'s `computeCircleTubeFillShapes` reuse the exact same per-cell hub+arm/passage-rectangle construction as the outline, emitting one closed, unrounded shape per cell/connector/stub instead of boundary lines ŌĆö see ADR 060; a crossing's weave marks are the one thing added as forced edges (arcs sealing the over tube across the under corridor's doors for a tangential-over crossing; two bridge sides jogging at mid-radius between the inward and outward door angles for a radial-over one), and corner rounding is a post-pass filleting every degree-2 line/arc vertex except inside crossing cells, where pieces are too short, or where another segment tip rests within the fillet's reach ŌĆö see ADR 059, and the closure-invariant regression test in `render.test.ts` (no dangling endpoint on generated mazes across sizes and seeds); the entrance is marked by a "hub" boundary around the center, sized directly from the maze's own starting `width` so its own passage width matches every other ring instead of a fixed, potentially-cramped radius ŌĆö with a gap at its own sector 0, mirroring the outer boundary's gap at the exit, and a single angle offset rotates both openings to point toward the top of the circle instead of the right; the solution trace follows the radius through a ring transition instead of cutting across at an angle (`computeCircleSolutionPoints`, see ADR 041), solution path computation (axis-aware at a crossing cell ŌĆö never turns from one passage onto the other, see ADR 024), PDF rendering (reMarkable 2 page size, with a visible opening at the entrance and exit ŌĆö see ADR 010), SVG image rendering for lightweight previews (see ADR 019), and reMarkable Cloud authentication + upload are implemented on top of the `rmapi-js` library (verified against a real reMarkable account ŌĆö see ADR 012, which supersedes the earlier hand-rolled protocol from ADR 007/008).
