# Circle-Crossing Tubes Still Visually Disconnected

## Description
The `circle-crossing` maze type's rendered tubes (`packages/core/src/circle-maze/render.ts`, `computeCircleTubeSegments`) still show visibly disconnected fragments ‚ÄĒ isolated "U"/hairpin/capsule shapes floating in open space, not connected to the rest of the maze ‚ÄĒ according to the user's own visual inspection (screenshots), even after three full redesign rounds in this same session, each independently verified by the previous agent via automated connected-component analysis and visual zoom inspection. This is a postmortem/handoff item: the user has asked to stop iterating and hand the problem to a different model (Fable 5) rather than continue the same investigation. Do not repeat the approaches already tried and rejected below ‚ÄĒ read the Notes section in full before starting.

## Acceptance Criteria
- [ ] The user visually confirms, on a real generated `circle-crossing` maze (any size/seed), that no tube boundary reads as a disconnected/floating fragment ‚ÄĒ verified against the user's own screenshots/zoom, not just the implementer's own review
- [ ] The fix (or diagnosis of what's actually being seen, if it turns out not to be a geometry bug) is written up in a new ADR under `.vibe/decisions/`, explaining what previous ADRs (055‚Äď058) got wrong that this one addresses
- [ ] Full test suite and lint stay green, including whatever new regression test(s) cover the specific case the user reports

## Notes

### Where the code lives
- `packages/core/src/circle-maze/render.ts` ‚ÄĒ `computeCircleTubeSegments` / `computeCircleCellTubeSegments`: the tube geometry itself. `computeCircleMazeSegments` in the same file is a *separate*, older thin-wall renderer for the plain (non-crossing) `circle` type ‚ÄĒ it has never had reported issues and may be worth comparing against as a working reference for how it handles ring-boundary walls ("drawn from the outer ring's own sectors, at their own angular resolution").
- `packages/core/src/circle-maze/render.test.ts` ‚ÄĒ current suite (30 tests, all green) for the tube renderer. Covers door-width consistency, no-angular-gap invariants, corner rounding, crossing gap behavior, and a "no dangling endpoints" check for one specific hand-built scenario ‚ÄĒ evidently insufficient, since it doesn't catch whatever the user is still seeing.
- `packages/core/src/circle-maze/cells.ts`, `topology.ts` ‚ÄĒ the underlying ring/sector graph (`carveEdge`, `outwardChildren`, `isCcwOpen`, `isInwardOpen`, `computeHubRadius`, `computeCircleSectorCounts`). Treated as stable/correct throughout this investigation; never identified as a bug source.
- `packages/core/src/rendering/maze-layout.ts` ‚ÄĒ the shared `rectangle-crossing` renderer (`computeTubeSegments`, `roundCorner`, `TUBE_HALF_WIDTH_RATIO`, `TUBE_CORNER_RADIUS_RATIO`), used throughout as the "known good" baseline for visual and structural comparison.
- `.vibe/decisions/055` through `058` ‚ÄĒ full ADR history of this feature and its rendering redesigns (055 and 058 currently `accepted`, 056 and 057 marked `superseded`).

### Everything tried, in order (none satisfied the user)
1. **ADR 055 original per-cell polar hub model**: small inset hub per cell in local (radius, angle) space, with an arm reaching to the cell boundary per open side. Bug found: door width/position inconsistent across ring transitions (the hub's angular half-width was sized from the ring's own angular step, `h * angleStep`).
2. **Fix**: door half-width resized to `hA = h / midRadius` (physical-radius-based, not angle-step-based). Fixed door-size consistency but not the structural issue found later.
3. **ADR 056 ‚ÄĒ full redesign to per-edge Cartesian channels**: every open connection becomes its own independent channel drawn directly between the two cells' real centers; closed sides reuse the separate thin-wall renderer (`computeCircleMazeSegments`) outright; a small hub-circle drawn at every node with ‚Č•1 channel. Rejected by the user on sight ‚ÄĒ "pourquoi je vois des ronds partout ?", "avec les perles, on ne peut plus faire le labyrinthe" (the hub circles read as a cluttered string of beads, not a maze).
4. Removed the hub circles (bare kinks at joints), then tried precise per-edge corner rounding using a general-angle tangent-circle fillet (`tangentDistance = radius / tan(angle / 2)`, since corners in the per-edge model aren't always 90¬į, unlike the rectangular grid). User rejected this too, after being challenged into doing a genuinely fair, matched-pixel-scale visual comparison against `rectangle-crossing`: the per-edge model structurally never gave closed sides a hub cap, so tube boundaries were visibly incomplete/open near every hub ‚ÄĒ reading as a wireframe, not a closed pipe.
5. **ADR 057 ‚ÄĒ reverted to the per-cell polar hub model** (attempts 1‚Äď2 above) plus corner rounding via the existing `roundCorner` (hub corners here are always radial‚ä•tangential, i.e. exactly 90¬į, so the same formula rectangle-crossing uses applies). Initially looked closed in the implementer's own review; user reported "les tubes ne sont pas ferm√©s" (the tubes aren't closed) twice more.
6. Root-caused via a **connected-components analysis** (union-find treating every segment endpoint as a graph node, merging on exact coordinate match) that an outward-side "margin cap" arc ‚ÄĒ the closed portion of a hub's boundary between its own corner and an off-center child's own door ‚ÄĒ only *passed through* that corner as an unmarked interior point of one long arc, instead of stopping there with a real shared vertex. Visually the arc's own sweep still covered the point (no literal pixel gap), but graph-wise it wasn't a real join. Fixed by splitting cap arcs exactly at the corner. Component count for a 14√ó14 test maze dropped from 434 to 55 (`rectangle-crossing`'s own baseline for a similarly-sized maze is ~28 components).
7. Found and fixed a **second, distinct bug** in the same per-cell-hub model: when a cell has *multiple* open outward children, its own inset hub corner can land *inside* one of their doors rather than in a capped margin ‚ÄĒ nothing exists there to visually close it off (the doorway must stay open). A closed cw/ccw wall reaching all the way to that corner anyway left a dangling, round-capped stub in the middle of an open passage ‚ÄĒ this is the reported hairpin shape. Fixed by retracting that wall to the hub's own mid-radius instead of reaching all the way out, whenever this specific situation is detected. The user then sent a **new** screenshot and said "c'est tout aussi mauvais" (just as bad) ‚ÄĒ still a visible disconnected "U": a diagonal double-line capsule with a rounded cap, not connected to anything.
8. User said **"change compl√®tement d'approche, c'est nul ce que tu fais"** (change the approach completely). Did a full architectural redesign, **ADR 058**: instead of every side anchoring to a *separately inset* hub window (`startHubA`/`endHubA`, narrower than the cell's own true angular width), every side now anchors directly to the cell's own **true** angular boundaries (`startAngle`/`endAngle`). Concretely:
   - Closed tangential (cw/ccw) side ‚Üí one plain radial line at the cell's own true boundary angle.
   - Open tangential side ‚Üí draws **nothing** ‚ÄĒ the same-ring neighbor's own inward/outward arcs already reach that exact same true-boundary point on their own, so they meet with no gap and no coordination needed between the two cells.
   - Radial (inward/outward) side ‚Üí always spans the cell's **entire** true angular width, capped everywhere except a door-sized notch per open connection (fanning to one notch per open outward child). Since `outwardChildren` always exactly partitions that same true width by construction (ADR 040), the very first and last pieces of this capped-arc sequence always land exactly on the cell's own true `startAngle`/`endAngle` ‚ÄĒ the same points the tangential sides anchor to ‚ÄĒ by construction rather than by a formula that has to be kept in sync by hand.
   - Also found and fixed a **third bug**, distinct from but structurally related to the previous two: the entrance (ring 0) and exit (last ring) cells were special-cased with a bare 2-line door and *no* margin capping, unlike every other open inward/outward side ‚ÄĒ meaning their own cw/ccw walls could dangle the same way attempt 7 fixed for interior cells. Unified them into the same general capped-door logic.
9. **Verified attempt 8** via: (a) connected-components analysis ‚ÄĒ dropped further to **14 components** for the same 14√ó14 test maze, *below* `rectangle-crossing`'s own ~28-component baseline, with the single largest component absorbing 4019 of ~4300 total segment endpoints; (b) direct visual inspection at multiple zoom levels of a real generated maze (14√ó14, seed 7), including the exact real-world cell (ring 1, sector 12 ‚ÄĒ two open outward children, closed ccw side) the originally reported hairpin came from ‚ÄĒ read as completely clean in the implementer's own review. **User's response: "cela ne fonctionne pas"** (it doesn't work) ‚ÄĒ gave up on further iteration and asked for this backlog item instead.

### Explicitly do NOT retry
- The per-edge Cartesian channel model (ADR 056) ‚ÄĒ rejected on visual grounds (cluttered "beads"), then found structurally incapable of producing closed hub boundaries. A dead end, explored in real depth.
- A separately-inset hub window, i.e. sizing a cell's own corners from `hA`/`startHubA`/`endHubA` independently of what its outward children need (ADR 057) ‚ÄĒ superseded specifically because independent computation is what caused every bug found in this investigation. Any redesign should preserve ADR 058's core insight (every side anchors to the cell's own *true* angular boundaries) unless a fundamentally different insight supersedes it.

### Open questions for whoever picks this up

The remaining ~12 (sizes reported as 210, 138, 56, 44, 33, 24, 20, 20, 8, 8, 8, 8, 4 segments) were never individually visualized ‚ÄĒ one of them may be exactly what the user is still seeing.
