## Modules
- [`modules/core.md`](modules/core.md) — shared core (maze generation pluggable across 4 selectable algorithms — growing-tree (default, difficulty-tunable, only one supporting bridge crossings), kruskal, wilson, aldous-broder — each isolated in its own module, and a choice of maze type — classic thin-wall `rectangle`, thick curvy-path `rectangle-crossing`, `circle` (a real growing-sector topology in its own `circle-maze/` module tree, entirely separate from the rectangular grid — more sectors per ring further out, always an exact multiple of the ring inside it so cells line up cleanly ring to ring, with its own generation, solver, and polar renderer), or `circle-crossing` (the polar equivalent of `rectangle-crossing`, combining the ring/sector topology with tube-style bridge crossings — see ADR 055; its tube rendering draws the outline of the maze's whole passage area built as a filled region of polar rectangles in a dedicated `region-boundary.ts` engine, closed by construction so disconnected tube fragments cannot exist, with the closed-pipe cap look of the rectangular type — see ADR 059), an optional path-length target (short/medium/long) that generates several candidate seeds (a user-configurable count, capped at 50) and keeps the best match, solution path and branch-point detection, PDF and SVG rendering sharing common wall/path-geometry layers, an optional light-gray tube background fill for the two tube types reusing the outline's own per-cell hub+arm/passage-rectangle construction as closed, unrounded shapes drawn under it (see ADR 060), SVG preview with an optional solution/branch-point overlay, generation-parameters footer, shared solution-mode/maze-type/algorithm/path-length validation, reMarkable Cloud auth + upload via `rmapi-js` isolated in its own `infrastructure/remarkable/` module boundary, verified against a real account)
- [`modules/cli.md`](modules/cli.md) — CLI (Commander) consuming `core` — `generate`, `send`, and combined `generate-and-send` commands implemented, with `--solution`, `--type`, `--algorithm`, `--path-length`, `--path-length-candidates` and `--tube-fill` options on `generate`/`generate-and-send`
- [`modules/web.md`](modules/web.md) — web server (Fastify) consuming `core` — a maze configuration form (including a maze type dropdown, a generation algorithm dropdown, a solution display mode dropdown for the PDF, a path length dropdown, a path-length candidate count field, a "show solution on preview" checkbox with a branch-point count display, and a "fill tube background" checkbox shown only for the two tube maze types — see ADR 060) with SVG image preview, PDF download, and direct send-to-reMarkable (with guided pairing and an optional target reMarkable Cloud folder) all implemented, with a dedicated stylesheet giving the page a responsive layout across mobile/tablet/wide-desktop breakpoints, including a two-column form/preview layout on wide screens (see ADR 054), and the form's last-used values remembered via a cookie and pre-filled on the next visit, and the difficulty/algorithm/candidate-count/tube-fill fields shown or hidden live based on the current maze type/algorithm/path-length selection so the user is never shown a field that isn't actionable (see ADR 053, ADR 060); `@fastify/helmet` security headers, and binds to `0.0.0.0` by default (override via `HOST`) — no auth layer, so anyone on the local network can reach it

## Observed patterns
- The web form's notion of which fields are conditionally relevant (`computeMazeFormFieldVisibility` in `packages/web/src/maze-form-field-visibility.ts`, mirrored in `public/app.js`) is derived from `core`'s own validation rules (`validateTypeAlgorithmCompatibility`, per-algorithm difficulty relevance, the path-length-candidate-count/path-length pairing) rather than hardcoded ad hoc in the frontend, the same "single source of truth mirrored client-side" pattern already used by `validateMazeFormInput`/`serializeFormPreferences` — see ADR 053
- A new maze type/algorithm combination is added by extending each of `core`'s three exhaustiveness-checked strategy/registry `Record`s (generation, solving, rendering) plus `maze-domain.ts`'s validation, one file per concern — demonstrated by `circle-crossing` (see ADR 055) reusing the exact same registration points `rectangle-crossing` and `circle` already established rather than inventing a parallel mechanism
- `core`'s domain vocabulary (`Maze`, `Cell`, `MazeType`, `MazeAlgorithm`, `PathLengthTarget`, `GenerateMazeOptions`, and their validation rules) is isolated in `maze-domain.ts`, depended on by generation/rendering code rather than the reverse, establishing the domain boundary noted as missing in `CLAUDE.md`'s `vibe:review-ddd` entry (see ADR 050); PDF rendering, SVG rendering, and their shared drawing geometry now live under their own `rendering/` module boundary within `core`, depending on the domain module but never depended on by it (see ADR 051); the reMarkable Cloud client now lives under its own `infrastructure/remarkable/` module boundary within `core`, marking it as an external-system adapter never depended on by the domain module (see ADR 052)
- Per-`MazeType`/`MazeAlgorithm` dispatch in `core` is centralized in `Record`-keyed strategy/registry maps (`maze-algorithm-registry.ts`, `rendering/maze-render-strategy.ts`, and a solver strategy map in `maze-solver.ts`) instead of duplicated conditionals, so TypeScript's exhaustiveness checking forces every new type/algorithm to be registered in one place per concern (see ADR 049)
- npm workspaces monorepo (`packages/*`), one `package.json` per package with `private: true`
- `core` resolves to its TypeScript sources for dev/test (via a `development` package export condition, no build required) and to its compiled `dist/` output for everything else, including the production web server and the packaged CLI binary (see ADR 044); a `postinstall` hook applies a `patch-package` fix for a broken ESM import in `rmapi-js` (see ADR 045)
- A Vitest test file next to each source module (`*.test.ts`)
- Style and lint centralized via Biome at the root (`biome.json`), tab formatting, double quotes
- Requires Node.js 25+ (pinned in `.nvmrc`, declared in the root `package.json` `engines` field) — needed by `rmapi-js`'s use of native `Uint8Array` base64/hex methods
