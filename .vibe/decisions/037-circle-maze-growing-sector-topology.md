# Circle maze: a real growing-sector topology, not a polar transposition of the rectangular grid

**Context:** ADR 034 (and its two follow-ups, ADR 035 and ADR 036) laid out "circle" as the exact same rectangular grid — a column reinterpreted as an angular sector, a row as a concentric ring — rendered in polar coordinates instead of Cartesian ones. With a fixed number of sectors at every ring, a sector's tangential (walkable) width at radius `r` is `r * angleStep`, which necessarily grows with radius for a constant angular step. No choice of ring radius (arithmetic, geometric, or bounded) can remove this: it can only trade off pinching passages near the center against ballooning them near the edge, or bound the ratio between the two — never make them genuinely uniform. Raised directly: the circle is not a transposition of the rectangle: it needs its own topology, not a reinterpretation of the rectangular grid's rows and columns.

**Decision:** Replace the polar-transposition approach with a real growing-sector topology (a "theta maze"): the innermost ring starts with `width` sectors; each successive ring's sector count doubles (or more generally grows by whatever factor keeps a cell's arc length from exceeding its radial thickness) as needed to keep cells roughly uniform in size at every radius, with ring radius growing arithmetically (constant radial thickness) throughout. Each outer-ring cell has exactly one inward neighbor; each inner-ring cell has one or more outward neighbors (however many outer sectors map onto it), computed with a proportional index mapping (`floor(sector * innerCount / outerCount)` for the inward direction, its exact inverse for the outward direction) so the mapping is consistent regardless of the exact growth factor between two rings. This is a genuinely different data model from the rectangular grid — cells no longer form a uniform 2D array, a given cell's neighbor count varies, and "north/south/east/west" no longer applies — so it needs its own cell representation, its own generation logic (still expressing the same 4 algorithms — growing-tree, kruskal, wilson, aldous-broder — but operating on this graph instead of a grid), its own solver, and its own rendering, entirely separate from the rectangular `rectangle`/`rectangle-crossing` code path, which is untouched by this decision.

**Reason:** This is the only way to keep passage width genuinely comparable from the center to the outer edge without an unavoidable trade-off — growing the sector count is what actually keeps a cell's tangential span bounded, instead of trying to compensate for a fixed span with the ring's radial thickness.

**Rejected alternatives:** Every radius-only adjustment tried under the polar-transposition approach (ADR 035's sector-count-based inner radius, ADR 036's geometric ring growth, and a further bounded-arithmetic compromise) — each one only moved the mismatch around (pinching vs. ballooning vs. a capped compromise) rather than removing it, because the underlying topology (fixed sectors per ring) can't produce uniform passages by construction. Reusing the rectangular algorithms' code by adding a generic graph abstraction they'd both share — considered, but changing already-shipped, tested rectangular-maze code to fit a shared abstraction was judged riskier than a self-contained parallel implementation for this new topology, consistent with the project's existing preference for isolated per-topology/per-algorithm modules (see ADR 033).
