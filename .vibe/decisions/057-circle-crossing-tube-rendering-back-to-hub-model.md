# circle-crossing tube rendering: back to a per-cell polar hub, with radius-based sizing and real corner rounding

**Context:** ADR 056 replaced the original per-cell polar hub (ADR 055) with per-edge Cartesian channels to fix door-alignment bugs. That redesign fixed alignment, but a fair, matched-scale visual comparison against `rectangle-crossing` (same pixel-per-cell-unit scale, not an optimistic crop) showed a real, substantive quality gap: closed sides reused the thin-wall renderer directly, with no hub cap connecting a node's channels — every junction and turn read as an incomplete, open wireframe rather than a closed pipe, unlike `rectangle-crossing`, which always traces one complete closed boundary per cell (arms *and* caps). A hub-circle patch tried in between (small fixed-radius circle at every multi-channel node) covered the gap but looked like a chain of beads and visually cluttered the maze, making it hard to trace.

**Decision:** Go back to a per-cell polar hub (the ADR 055 shape: a small inset window in local radius/angle space, with an arm reaching to the cell boundary for each open side, a flat cap otherwise) instead of per-edge channels, and fix the two problems that motivated ADR 056 directly at the hub level rather than abandoning the shape:
- A door's angular half-width is derived from the physical radius (`half-width / mid-radius`) instead of the ring's own angular step, so its physical (Cartesian) size stays consistent across rings regardless of how many sectors a ring happens to have, or how deep in the maze it sits.
- The outward side sizes/positions each open child's door from *that child's own* angle and radius (not the parent's), so it always agrees exactly with the same door as drawn by the child's own inward side — this was already fixed under ADR 055 and carries over unchanged.
- Every hub corner where its two adjacent sides share the same open/closed state gets rounded with the existing rectangular `roundCorner` fillet, exactly as `rectangle-crossing` already does — a hub corner here is always between one radial-direction side and one tangential-direction side of the same cell, which are inherently perpendicular, so the same 90°-only formula applies correctly. This restores the complete-closed-pipe look ADR 056 lost, without the per-edge model's alignment fragility or the hub-circle patch's visual clutter.

**Reason:** The per-cell hub was never the wrong *shape* — every follow-up bug (door misalignment, within-band width drift) came from deriving a shared piece of geometry (a door's width and position) independently on two different cells instead of from one consistent rule. Both of those are now fixed as formulas *within* the hub model (radius-based sizing, child-owned door position) rather than requiring a different model altogether. The per-edge redesign traded that fragility for a structural gap (no hub cap) that turned out to be worse for the maze's actual purpose — reading and tracing it — since a real 2D printed maze depends on solid, unambiguous boundaries at every junction. Confirmed via an honest, matched-pixel-scale visual comparison against `rectangle-crossing` at multiple randomly chosen crops (hub-dense area near the center, sparse area near the outer boundary, and a bridge crossing), all showing closed tubes, consistent corner rounding, and correct crossing gaps.

**Rejected alternatives:**
- Keeping the per-edge Cartesian channel model and adding hub caps back onto it (a hybrid): rejected as unnecessary complexity — once caps are added back, the per-edge model degenerates back into essentially the same shape as the per-cell hub, without any of the alignment benefits the per-edge model was meant to provide (since the caps still need cell-local, not edge-local, angular bookkeeping).
- The hub-circle patch (ADR 056's own gap fix, a fixed-radius circle at every multi-channel node): rejected by the user as visually cluttering — it reads as a chain of connected beads rather than a maze, making the path harder to trace.
- A general-angle corner fillet (`radius / tan(angle / 2)`) applied to the per-edge model's arbitrary-angle corners: mathematically correct for its narrow scope, but built on top of the same structurally incomplete per-edge model, so it did not by itself fix the missing-hub-cap defect and became moot once the per-edge model itself was abandoned.
