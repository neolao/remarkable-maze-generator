# Circle maze type as a polar rendering of the same rectangular grid

**Context:** Backlog item 027 asked for a "circle" maze type. A literal circular maze — rings holding progressively more cells the further out they are, merging inward — would require its own cell/wall model, its own solver, and its own generation logic entirely separate from the existing rectangular grid (`Cell[][]`/`CellWalls`, north/south/east/west), a scope far larger than the previous `rectangle-crossing` type addition, which only changed rendering on top of the same grid.

**Decision:** `circle` reuses the exact same rectangular grid, generation algorithms (`growing-tree`, `kruskal`, `wilson`, `aldous-broder`), difficulty option, solver, and branch-point detection as `rectangle`/`rectangle-crossing`, entirely unchanged — a column index becomes an angular sector, a row index becomes a concentric ring (row 0 innermost, last row outermost). The one generation-level change: the grid wraps horizontally end-to-end (the last sector's east neighbor is the first sector, and vice versa) so the maze reads as a real, seamless circle instead of a pie with one slice always walled off from its neighbor — implemented as a per-generation topology flag consumed by all 4 algorithms' neighbor lookups (`x` wraps via modulo instead of being bounds-excluded), not a new pattern each algorithm invents independently. Entrance stays the first cell (center of the innermost ring), exit stays the last cell (outermost ring), mirroring the existing top-left/bottom-right convention. Rendering gets a new polar geometry: ring-boundary walls become arcs, sector-boundary walls become radial line segments — computed by a new function parallel to the existing wall/tube segment computations, consumed by both the PDF and SVG renderers.

**Reason:** Keeping the same grid, solver, and 4 algorithms means `circle` costs roughly what `rectangle-crossing` cost (a generation-topology tweak plus a new rendering path), not a second maze engine. The wraparound is done at the generation level (not stitched on after the fact) so the single-solution/fully-connected guarantee holds exactly as it does for the other two types — forcing the seam open after generation without considering it during carving could create a second path around the ring, breaking that guarantee.

**Rejected alternatives:** A literal circular maze with a variable number of cells per ring — closer to what a real circular hedge/corn maze looks like, but requires an entirely new cell/wall representation, a new solver, and new generation logic; too large for this item, and left as a possible future item if this lighter version turns out not to be enough. No horizontal wraparound (a fixed radial seam wall between the last and first sector on every ring) — simpler, but reads visually as a pie chart with a permanent slice boundary rather than a real circle, undermining the point of the feature.
