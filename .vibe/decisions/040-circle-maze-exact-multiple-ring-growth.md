# Circle maze: sector counts grow by an exact integer multiple between rings
**Context:** ADR 038 computed each ring's sector count directly from that ring's own actual radius, independently of the previous ring's count, to keep passage width consistent. ADR 039 then restricted ring-to-ring passages to a sparse set of "portal" sectors, to compensate for the resulting incoherence: since a ring's count was no longer required to be a multiple of the ring inside it, an inner cell could have many outward children spread unevenly across its own angular span, and whichever one carved an opening could land anywhere inside that span — including its middle — with no visual relationship to the parent's own boundaries. Cross-referencing the reference implementation at github.com/codebox/maze.js confirmed the root cause: its `cellCountsForLayers` always grows the sector count by a rounded integer ratio (an exact multiple), which is exactly the property ADR 038 dropped.
**Decision:** Revert to growing each ring's sector count by multiplying the previous ring's count by a rounded integer ratio — the same technique ADR 037 originally used, and the one the reference implementation uses — evaluated at that ring's actual radius (`ring index + computeHubRadius(width)`, keeping ADR 038's width-derived hub sizing). Ring 0 still has exactly `width` sectors. Because every ring's count is now guaranteed to be an exact multiple of the ring inside it, `inwardParent`/`outwardChildren`'s proportional mapping always splits a parent's angular span into an exact whole number of equal-sized children, so a child's boundary always lines up exactly with either the parent's own start angle or an exact fraction of its span — never an arbitrary rounded position. This removes the need for ADR 039's portal restriction, which is dropped along with it: `neighborsOf` goes back to allowing a connection at any sector, since the alignment problem it was compensating for no longer exists at the source.
**Reason:** Fixing the actual cause (the growth ratio) is simpler and more robust than restricting which positions are allowed to open (the symptom), and matches a proven, working reference design instead of an invented compensating mechanism.
**Rejected alternatives:** Keep both the independent per-ring formula and the portal restriction — this treats the symptom rather than the cause, and still allows scattered, incoherent openings within whichever sectors remain eligible. Increasing `MIN_PORTAL_SPACING` further — same objection, plus an additional free parameter with no principled value.
