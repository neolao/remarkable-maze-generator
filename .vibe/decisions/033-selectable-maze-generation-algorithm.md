# Selectable maze generation algorithm

**Context:** Maze generation used a single fixed technique (the growing-tree algorithm, see ADR 015), with no name exposed to callers and no way to pick a different technique. Backlog item 026 asked for a spike prototyping alternative algorithms; based on early results the scope was expanded to exposing them as real, user-selectable options in the CLI and web UI, rather than staying as unshipped research code.

**Decision:** Introduce an `algorithm` option alongside the existing `type` option, following the exact same shape (a string union, a validity list, a checker function, an "invalid value" message helper). The current technique is named `"growing-tree"` and stays the default, with byte-for-byte identical output for existing (unspecified-algorithm) callers. Three additional algorithms are added as real, selectable options: `"kruskal"` (randomized Kruskal's, union-find over shuffled edges), `"wilson"` (loop-erased random walk / true uniform spanning tree), and `"aldous-broder"` (random walk over all cells / true uniform spanning tree, simpler but slower). Each algorithm is implemented in its own isolated module producing a wall grid, with no shared mutable state or shared tuning logic between them, so any one of them can be iterated on or re-tuned later without risking a behavior change in the others. Two limitations apply for this iteration: the `difficulty` knob (1–5) only affects `"growing-tree"` — the other three ignore it, since none of them has an equivalent selection-bias lever yet — and the `"rectangle-crossing"` (bridge) maze type stays exclusive to `"growing-tree"`, since bridge tunneling is carved as part of that algorithm's own traversal; requesting it with any other algorithm is rejected with a clear validation error, the same way an invalid `type` or `algorithm` value is rejected.

**Reason:** Mirroring the existing `type` option's shape keeps the codebase's two "pick one of a few named variants" concepts (`type`, `algorithm`) consistent and equally familiar to maintain. Isolating each algorithm in its own module directly serves the stated goal of being able to iterate on one algorithm's efficiency/tuning independently, without one algorithm's changes silently affecting another's behavior or tests. Deferring a difficulty-equivalent and bridge-crossing support for the three new algorithms keeps this iteration scoped to "selectable and correct", leaving "as tunable/as feature-complete as growing-tree" as a deliberate follow-up once real usage tells us which of the three is worth investing further in.

**Rejected alternatives:** Folding algorithm choice into the existing `type` option (e.g. `"rectangle-kruskal"`) — rejected because `type` already carries an orthogonal concern (bridge crossings vs. plain corridors) and conflating the two would force an exponential blow-up of type names as algorithms are added/removed. Implementing a best-effort difficulty mapping for the new algorithms right away (e.g. biasing Kruskal's edge shuffle) — rejected as premature: each algorithm's natural tuning knob is different enough that guessing one now, before any of them is chosen for further investment, risks locking in a poor mapping.
