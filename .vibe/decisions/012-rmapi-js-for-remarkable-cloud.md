# Use rmapi-js instead of a hand-rolled reMarkable Cloud protocol

**Context:** The hand-rolled reMarkable Cloud protocol from ADR 007/008 (raw `fetch` calls to `webapp.cloud.remarkable.com` and `document-storage.cloud.remarkable.com`) was implemented against community documentation that turned out to be obsolete: `document-storage.cloud.remarkable.com` no longer resolves at all. reMarkable has moved to a newer, more complex "sync15" protocol (content-addressed blob storage with root-hash generations). This was discovered when a real upload attempt failed with a raw, unwrapped `fetch failed` error — itself a secondary bug, since `uploadPdf`'s network calls, unlike `remarkable-auth.ts`'s, weren't wrapped to produce a clear message.

**Decision:**
- Replace the hand-rolled protocol with [`rmapi-js`](https://www.npmjs.com/package/rmapi-js) (MIT, actively maintained), which implements the current sync15 protocol correctly.
- `authenticate(store, pairingCode, options?)` now wraps `rmapi-js`'s `register()` (pairing code → device token, persisted via `CredentialStore` exactly as before — its interface is unchanged) and `remarkable()` (device token → ready API client). Both are wrapped in `try`/`catch` to produce clear, consistent error messages (including detecting `ResponseError` with a 400/401 status specifically as an invalid/expired pairing code).
- `RemarkableSession` is now an **opaque alias** for `rmapi-js`'s `RemarkableApi` client instance, not a `{ deviceToken, userToken }` data object. Callers (CLI, and any future consumer) only ever receive it from `authenticate()` and pass it to `uploadPdf()` — they never construct or inspect it. This is the abstraction the user asked for: reMarkable Cloud protocol details are fully contained in `core`.
- `uploadPdf(session, filePath, visibleName, options)` now delegates to `session.uploadPdf(visibleName, bytes)`, wrapped in `try`/`catch` for a clear error message. The manual ZIP-building code and the `fflate` dependency are removed entirely — `rmapi-js` handles the wire format.
- `rmapi-js` requires `Uint8Array.prototype.toBase64()`/`fromHex()` etc. (TC39 proposal-arraybuffer-base64), which are only native starting in **Node.js 25** — not Node 22 as previously documented. The project's minimum Node version is raised accordingly: `.nvmrc` pinned to `25`, `engines.node` set to `>=25` in the root `package.json`, README/CLAUDE.md updated. This was a deliberate choice over adding a polyfill, confirmed with the user.
- The CLI's `send.ts` drops its `fetch`/`baseUrl` passthrough options entirely — there was no equivalent single "base URL" concept in `rmapi-js` (it has separate `authHost`/`uploadHost`/`rawHost`), and none of that needs to be exposed at the CLI layer.
- Tests no longer spin up a fake HTTP server mimicking the reMarkable wire protocol (that protocol is now `rmapi-js`'s responsibility, and it's a maintained library, not ours to re-verify). Instead: `remarkable-auth.test.ts` uses `vi.mock("rmapi-js")` to test our wrapping/orchestration logic; `remarkable-upload.test.ts` and `send.test.ts` pass/mock a fake session object directly.
- The fix was verified with a **real upload to a real reMarkable account** (the user had already paired a real device), not just mocks — confirming the new protocol genuinely works, not just that our code calls the right functions.
- Vitest needed `test.server.deps.inline: [/rmapi-js/, /crc-32/]` in `vitest.config.ts` to resolve correctly; without it, a transitive dependency (`crc-32`) fails to resolve an extensionless subpath import under Vitest's default externalized-SSR-deps handling, even though it resolves fine under plain Node.

**Reason:** Protocol correctness for an unofficial, frequently-changing API is exactly the kind of problem better delegated to an actively maintained library than re-implemented and re-verified in-house. The opaque `RemarkableSession` type keeps `core`'s public API stable even if `rmapi-js`'s internals change again. Raising the Node requirement was preferred by the user over polyfilling a very recent language feature.

**Rejected alternatives:** Patching the existing hand-rolled protocol to point at correct current endpoints — rejected because the actual current protocol (sync15) is materially more complex (content hashing, generation-conflict retries) than the old one, making a from-scratch reimplementation high-risk and high-maintenance compared to adopting a maintained library. Polyfilling `Uint8Array.prototype.toBase64`/`fromHex` to stay on Node 22 — rejected per user preference to raise the Node requirement instead.
