# circle-crossing tube rendering: anchor every side to this cell's own true angular boundaries, not a separately inset hub window

**Context:** ADR 057's per-cell hub (a small window inset both radially *and* angularly, via `startHubA`/`endHubA` narrower than the cell's own true `[startAngle, endAngle]`) fixed the visible gaps found at the time, but kept producing new disconnected-tube instances as more of the maze was inspected: an outward margin-cap arc could span across this cell's own inset corner without stopping there (a real vertex was needed, not just visual overlap); a cell with several open outward children could have its own inset corner land *inside* one of their doors, leaving a cw/ccw closed wall dangling into open space with a round cap — the reported "impossible" isolated hairpin shape. Both bugs, and the general pattern of continuing to find new ones, traced back to the same root cause: this cell's own hub corners were computed independently of what its outward children actually needed, so the two only lined up by coincidence.

**Decision:** Anchor every side directly to this cell's own **true** angular boundaries (`startAngle`/`endAngle`) instead of a separately inset window:
- `cwSegment`/`ccwSegment` (tangential): closed is one plain radial line at this cell's own true boundary angle; open draws *nothing* — the same-ring neighbor's own inward/outward arcs already reach that exact same boundary point on their own (same radius, same angle), so they meet with no gap and no coordination needed between the two cells.
- `inwardSide`/`outwardSide` (radial): always span this cell's own entire true `[startAngle, endAngle]` at `innerHubR`/`outerHubR`, capped everywhere except a door-sized notch centered on each real open connection (fanning out to one notch per open child on the outward side, exactly as before). Since `outwardChildren` always exactly partitions that same true width (ADR 040), the very first and last pieces always land exactly on this cell's own `startAngle`/`endAngle` — the same points `cwSegment`/`ccwSegment` anchor to — by construction, not by a formula that needs to be kept in sync by hand.
- The entrance and exit cells, previously special-cased with a bare 2-line door and no margin caps, now go through the exact same capped-door logic as every other open inward/outward side (this was itself a second, distinct bug — the entrance/exit special case had silently never gotten the margin-capping fix applied to the general case).
- The 4 hub corners are now simply `(innerHubR/outerHubR) × (startAngle/endAngle)`, rounding the same way as before wherever their two adjacent sides share the same open/closed state.
- Crossings: tangential-over draws its two full-width true-boundary arcs unconditionally (the same shape a plain closed cell's own arcs already have); radial-over bridges straight through the hub's own interior at this cell's own door angle, then hands off to `outwardSide`'s own child-aware positioning from `outerHubR` onward.

**Reason:** Once every side agrees on the same reference points by construction, the specific angular relationship between a cell and its children stops mattering for closure — there is no longer a separate "hub window" position that can fall out of step with where the outward children's own doors happen to be. Verified two ways: (1) a connected-components analysis of the rendered geometry (treating every segment as a graph edge, coordinates as nodes) — `rectangle-crossing` produces ~28 components for a 14×14 maze; the ADR 057 hub-inset model produced 434; this design produces 14, with the single largest component absorbing 4019 of ~4300 total segment endpoints; (2) direct visual inspection at multiple zoom levels of a real generated maze, including the specific real-world cell (two open outward children, closed ccw side) the reported hairpin came from.

**Rejected alternatives:**
- Continuing to patch the ADR 057 inset-hub model (e.g. splitting cap arcs at the inset corner, retracting a wall that would dangle into an open door): rejected after the second and third patch each fixed one concrete instance while leaving the same underlying mismatch in place for the next one found — a genuine design-level fix rather than another special case, per explicit user direction ("change complètement d'approche").
- Widening only the *outward*-facing edge of the hub to the cell's true boundaries while keeping the inward-facing edge inset (a trapezoid hub): would have fixed the outward-side fan-out mismatch but not the entrance/exit special-case bug (a separate defect, unrelated to fan-out), and keeps two different reference systems (inset vs. true) alive in the same cell for no benefit once the inward side gets the same fix anyway.
