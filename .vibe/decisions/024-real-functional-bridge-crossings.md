# Bridge crossings are real, walkable passages, not decorative stubs

**Context:** ADR 022 made the `rectangle-crossing` maze type's "under" passage purely decorative: a visual stub with no real connectivity, chosen specifically to avoid the risk of a cycle breaking the maze's "exactly one solution" guarantee. The user rejected this outright: a maze must never show a corridor that looks walkable but isn't reachable — "il ne faut pas créer de chemins inatteignables," a rule that applies to any maze type, not just this one.

The user pointed to David Bau's original generator (`pypdfmaze`, https://github.com/akaihola/pypdfmaze/blob/master/pymaze.py) as reference: its `canbuild`/`dobuild`/`cancross` functions let a new corridor "tunnel" through an already-built straight passage to reach new territory, creating a real, structurally-connected crossing. One specific line (`maze[pos] ^= random.choice((0, 15))`) has bit-level semantics that stayed ambiguous even after careful reading, and the script (Python 2, ReportLab, CGI-era) could not be run in this environment to verify empirically. Rather than port code whose exact behavior couldn't be confirmed, the user agreed to a clean, independently-specified re-design of the same underlying idea.

**Decision:** For `type: "rectangle-crossing"`, extend the existing growing-tree generation (ADR 015) — unchanged for `rectangle` — with a second kind of move: alongside "extend to an adjacent unvisited cell," a cell that is currently a clean straight passage (both walls open on one axis, both closed on the other) and does not already host a crossing can be **tunneled through**, in the perpendicular direction, to reach a new unvisited cell one step beyond it. Both the normal move and the tunnel move are eligible candidates in the same random selection, so `difficulty` continues to control branch density identically for both maze types.

Tunneling opens all 4 walls of the tunneled-through cell (both axes now genuinely connect to their respective neighbors) and records a `MazeCrossing { x, y, underAxis }`, where `underAxis` names the pre-existing axis purely as a rendering hint (which passage paints as going "under"). Because every edge — direct or tunneled — always connects to a cell that was previously unvisited, the whole structure stays a tree: exactly one path exists between any two cells, entrance and exit included, with no risk of an accidental shortcut cycle.

Both axes at a crossing cell are equally real and walkable. `solveMaze` (`maze-solver.ts`) is extended so a path may continue along the axis it entered a crossing cell on, but never turn onto the other axis at that same cell — modeled by splitting a crossing cell into two graph nodes (one per axis) for traversal purposes, without changing the public `MazePosition` shape.

Rendering (`maze-layout.ts`) mirrors this, with one important refinement found after a first attempt: `computePathSegments` excludes *only* the over-axis connection at a crossing cell — the under-axis connection stays a perfectly ordinary segment, drawn in the same pass as every other corridor in the maze. `computeCrossingOverSegments` then emits just the over-axis pair (2 segments per crossing, reaching that axis's own neighbor centers), painted afterward, on top. The first version drew *both* axes as a dedicated, separately-painted "under"/"over" pair reaching all the way to their neighbor centers; because the "under" pair was painted before *every* other corridor in the maze (not just its own crossing's "over" pair), it could be accidentally cut by an unrelated corridor elsewhere that happened to physically overlap near a neighboring cell's center (visible as a stray notch or an isolated floating dash — reported by the user from a rendered maze). Confining the special-cased paint-order interaction to exactly the 2 over-axis segments of each crossing, and letting the under-axis behave like any other corridor, eliminates that: a crossing's "over" tube can now only ever cut its own "under" tube, never anything else.

**Reason:** A generated maze must never present a corridor a solver could follow into a dead end that was never actually connected — this is a correctness property expected of every maze type, not a stylistic choice specific to crossings. Extending the same growing-tree/spanning-tree construction that already guarantees connectivity and a unique solution (rather than opening extra walls in a second post-processing pass over an already-completed tree, which would create real cycles and put the single-solution guarantee at risk) was the only way to add real crossings without reopening that guarantee to question.

**Verification:** in addition to the existing "every cell reachable" check (`countReachableCells`, ignoring axis), `maze.test.ts` adds `findCellsReachableByATraceablePath` — an axis-aware BFS mirroring `solveMaze`'s own traversal rules — asserting a real, non-turning path can be traced from the entrance to every single cell, for both maze types. `maze-solver.test.ts` adds hand-built cases proving the solver both finds a valid same-axis route through a crossing and rejects a route that would require turning between axes at one, even when every individual wall along the way is open.

**Rejected alternatives:**
- Byte-for-byte porting `pypdfmaze`'s bit-flag tunneling (including the ambiguous XOR trick) — rejected: could not be verified without running the original Python 2 script, and shipping unverified, misunderstood logic risked a subtle correctness bug more consequential than the decorative-stub problem it would replace.
- Opening extra walls in a post-processing pass after the spanning tree is already fully built — rejected: since every cell is already connected to every other cell through some tree path by the time generation finishes, any additional wall-opening between two cells not yet directly joined creates a real cycle, which can introduce a second entrance-to-exit route depending on where it falls — incompatible with the maze's "exactly one solution" requirement.

**Follow-up fix — no adjacent crossings:** The first working version could legally place several separate crossings on immediately adjacent cells, tunneling repeatedly between the same two parallel corridors. Each one was individually valid (still a real, walkable, single-solution-preserving tree edge), but visually the stacked result read as a repeating "ladder" between two corridors rather than an occasional, notable bridge — reported by the user from a rendered maze. `generateMaze` now also rejects a tunnel candidate whose intermediate cell would be orthogonally adjacent (Manhattan distance 1) to an already-placed crossing, keeping crossings visually isolated. Covered by a new test asserting no two crossings in a generated maze are ever adjacent.
