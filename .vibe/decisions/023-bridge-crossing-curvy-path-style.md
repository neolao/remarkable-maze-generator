# Bridge crossing maze type: thick rounded-path rendering, not thin walls

**Context:** After the initial `rectangle-crossing` implementation (ADR 022) rendered crossings as small perpendicular ticks over thin wall lines, the user compared it against the visual style of David Bau's "Printable Mazes" (the original inspiration) and found it unconvincing ã the reference style draws mazes as thick, rounded-cap corridor paths ("tubes"), not thin wall boundaries, which reads far more clearly for a crossing (a tube visibly passing over/under another).

**Decision:** For `type: "rectangle-crossing"` only, replace wall-based rendering with corridor-centerline rendering: a thick stroke, round line caps, following the path graph (`computeWallSegments` ã `computePathSegments`, plus `computeCrossingBridgeSegments` for crossings). The centerline connects open-passage cell centers, plus a stub from the entrance/exit cell center to the boundary. At a crossing cell, the real ("over") passage is drawn as an uninterrupted thick segment; the decorative ("under") passage is two short segments confined to the crossing cell itself, with a gap in the middle, drawn *before* the "over" segments so the real passage always paints on top. The `rectangle` type is untouched ã still thin wall lines, unaffected by this change (confirmed by its unmodified regression tests continuing to pass).

A first pass used a 40%-of-cell-size stroke, guessed without a ground truth reference ã the result read as thick black blobs with no visible crossing effect (user feedback). The user then pointed to David Bau's original Python implementation (`pypdfmaze`, https://github.com/akaihola/pypdfmaze/blob/master/pymaze.py), which exposes the real ratios: line width `wall=0.3` of the *half*-cell (i.e. 0.15 of the full cell), and a "tube" reach of `tube=0.7` of the half-cell for each path arm. Adopting the 0.15 line-width ratio (`PATH_THICKNESS_RATIO`, the single source of truth shared by `maze-layout.ts` and both renderers) immediately produced a legible tube-and-crossing look; the crossing stub geometry (confined to the crossing cell, gap in the middle) turned out to already be structurally equivalent to the reference's own crossing cue once the thickness was fixed ã the thickness, not the stub shape, was the actual bug.

**Reason:** A crossing is only legible as "one path going over another" when the main path is a visibly clean, continuous tube and the "under" stub's gap is a small fraction of a *thin* line ã at 40% thickness the gap was swallowed by the stroke width itself. Restricting the change to `rectangle-crossing` avoids altering the appearance of every existing maze and keeps the two types visually distinct, matching the user's explicit choice to scope the redesign. Grounding the ratio in the actual reference implementation (rather than eyeballing a guess) is what fixed it: see [[feedback_visual_style_verification]] in memory.

**Rejected alternatives:**
- Applying the thick rounded-path style to both maze types ã rejected: would change the appearance of the already-shipped classic maze type for users who did not ask for it.
- A byte-for-byte replica of David Bau's renderer (its per-cell arc/rotation path router, `curve=1` quarter-circle arcs at every turn) ã rejected as disproportionate; straight centerline segments with round line caps already produce a convincingly curvy, clearly-legible tube look with the existing line-drawing primitives (pdf-lib `drawLine` with `LineCapStyle.Round`, SVG `stroke-linecap="round"`), no new geometry primitives (arcs/beziers) needed.
- Extending the "under" crossing stub a full cell into each neighboring cell (tried during this same investigation, before finding the reference source) ã rejected: not what the reference does, and it risks visually overlapping unrelated corridors in the neighboring cells.
