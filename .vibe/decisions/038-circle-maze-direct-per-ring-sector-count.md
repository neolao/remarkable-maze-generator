# Circle maze: direct per-ring sector count and width-derived hub radius
**Context:** ADR 037 grew each ring's sector count by multiplying the previous ring's count by a rounded integer ratio, and the entrance "hub" boundary around the center used a small fixed radius (0.3, later a fixed full unit). Passage width (the arc length of a wall or opening) ended up swinging widely (~0.35 to ~2.1 units) because rounding error compounded ring over ring, and the hub radius wasn't tied to the maze's own starting `width`, so a small starting `width` produced a cramped, barely-visible entrance while a large `width` could overflow its own hub.
**Decision:** Compute every ring's sector count directly from that ring's own actual radius (`ring index + computeHubRadius(width)`) instead of multiplying the previous ring's count. Derive the hub radius directly from `width` — solving `2·π·radius / width = RADIAL_THICKNESS` for `radius` — clamped to a minimum of one full radial-thickness unit so the entrance is never smaller than any other ring.
**Reason:** `inwardParent`/`outwardChildren`'s proportional index mapping already tolerates any pair of ring counts, not just exact multiples, so nothing requires the multiplicative growth. Computing each ring independently keeps every cell's arc length tightly close to the radial thickness (tested within roughly ±15%, down from the old ±110% swing), and tying the hub radius to `width` keeps the entrance's own passage width consistent with the rest of the maze regardless of how many sectors it starts with.
**Rejected alternatives:** Keep the multiplicative growth but only enlarge the fixed hub radius — this does not fix the compounding rounding error ring to ring, and a fixed hub radius still doesn't scale with `width`. Increasing precision of the rounding (e.g. keeping a fractional running count) was considered but adds complexity for no benefit once the direct per-ring formula is available.
