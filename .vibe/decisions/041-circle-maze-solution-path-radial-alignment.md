# Circle maze: solution path follows the radius through a ring transition
**Context:** The solution trace was drawn as a straight line directly between each pair of consecutive cells' own centers. Within a single ring this is fine, but across a ring transition the two cells are frequently at different angles (whenever the outer cell isn't the first of its parent's several children), so the line cut across the maze at a visible diagonal instead of appearing to follow a passage between the two rings.
**Decision:** Introduce `computeCircleSolutionPoints`, which expands a raw solution path into the actual points to draw: whenever consecutive path cells are on different rings, it inserts an extra point at the shared ring boundary, at the *outer* cell's own angle. The segment from that point into the outer cell's center is then a pure radial line (same angle, only the radius changes). Both the SVG and PDF renderers now call this shared function instead of independently drawing a line straight between two raw cell centers, so the two stay in sync.
**Reason:** A real passage between two rings, drawn on paper, is understood to run along the radius, and the wall geometry itself already reflects that (it draws each ring's own boundary using that ring's own angle). The solution trace should read the same way. Anchoring the inserted point to the outer cell's own angle (rather than, say, the midpoint between the two cells' angles) keeps the fix simple and cheap to compute, and fully removes the diagonal for the segment that matters most (the approach into the passage itself); the short segment from the inner cell's own center to that point can still be a slight diagonal, but it stays contained within that single inner cell instead of spanning the visible gap between rings.
**Rejected alternatives:** Replicating the reference implementation's (github.com/codebox/maze.js) full branch-by-branch solution-path rendering, which special-cases every combination of incoming/outgoing direction at a cell with multiple children to draw an exact arc-plus-radial-line route — more precise, but a large increase in complexity for a marginal visual improvement over anchoring at the outer cell's own angle.
