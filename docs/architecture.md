# Architecture

remarkable-maze-generator is organized into three main parts, grouped in a single repository (monorepo):

- **The core (`core`)** — maze generation, PDF layout, and the connection to the reMarkable Cloud API for sending documents to the tablet all live here. This core is shared: business logic is written only once.
- **The CLI** — a command-line interface that relies on the core to generate a maze and produce the corresponding PDF from a terminal.
- **The web UI** — a web server that also relies on the core, serving both a visual interface (web page) and an API to generate mazes from a browser.

## How the parts communicate

```
        ┌───────────────┐
        │     core      │   maze generation, PDF, reMarkable Cloud
        └───────┬───────┘
        ┌───────┴───────┐
        ▼               ▼
 ┌─────────────┐  ┌─────────────┐
 │     CLI     │  │   Web UI    │
 └─────────────┘  └─────────────┘
```

The CLI and the web UI contain no generation logic of their own: they call the core and simply expose that logic through a terminal or a browser. This guarantees that a generated maze produces the same result whether it comes from the CLI or the web app.

## Current status (v0.5.0)

- **Core**: maze generation, automatic solution solving, PDF export, and image (SVG) export are all implemented. Generated mazes are sized for the reMarkable 2 tablet and show a clearly visible entrance and exit opening, with an optional solution page or overlay. A difficulty level (1 to 5) controls how many branch points and decision points the maze has, from easy (long winding corridors) to hard (many branch points); dead-end branches stay meaningfully long to explore at every difficulty level, instead of being cut short. Two maze types are available: the classic type (thin wall lines), and a "bridge crossing" type drawn as a real hollow tube with wide corridors and smoothly rounded corners (two independent edge lines per corridor) where some real, walkable corridors visibly duck underneath one another, while still keeping exactly one solution and every corridor genuinely reachable. Every generated PDF prints the exact parameters used to create that maze (type, dimensions, seed, difficulty) in a small footer, so it can be regenerated identically later. Several mazes can be generated as a batch, either as one combined PDF or as separate files. reMarkable Cloud authentication and PDF upload (including sending into a specific existing folder) are implemented and have been verified against a real reMarkable account.
- **CLI**: mazes can be generated and saved as a PDF from the terminal, with an optional difficulty level and an optional solution display mode (none, extra page, or overlay) (also reachable via the `generate.sh` shortcut at the repository root), and a local PDF can be sent straight to a reMarkable Cloud account — optionally into a named folder — with a guided one-time pairing prompt on first use (also reachable via `send.sh`). A command combining generation and sending in a single step is also available (also reachable via `generate-and-send.sh`), keeping the local PDF and reporting a clear error if the upload fails after generation succeeds.
- **Web UI**: a maze can be generated through a web API endpoint (accepting size, seed, difficulty, maze type, and solution display mode, returning the PDF directly) and through a web page form that shows an image preview right after submission, with a link to download the matching PDF, and invalid input caught before any request is sent. The form also lets a user pick the solution display mode (none, extra page, or overlay) for that PDF, applied consistently to both the download and the reMarkable Cloud upload. The preview can separately, optionally trace the maze's solution directly on the image, marking every branch point along the path with a circle and showing the total branch point count. The web page can also send a maze directly to a reMarkable Cloud account, reusing the same pairing as the CLI; if the browser isn't paired yet, a guided one-time pairing step appears and the send retries automatically once pairing succeeds.

## Requirements

Node.js 25+ is required, mainly for native `Uint8Array` base64/hex support used by the reMarkable Cloud integration (pinned in `.nvmrc`).
